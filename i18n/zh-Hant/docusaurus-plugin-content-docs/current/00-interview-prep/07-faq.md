## General

### 什麼是 AI 工程師？

AI 工程師在大型語言模型之上打造生產系統。這個角色介於傳統軟體工程與機器學習研究之間：較少做模型訓練，較多圍繞既有模型做系統設計。日常工作涵蓋提示與上下文工程、檢索管線、代理迴圈、評估框架，以及維持整套系統上線的基礎設施。請參閱 [AI 就業市場趨勢](06-job-market-trends-2026.md)。

### AI 工程師與 ML 工程師有什麼差別？

ML 工程師訓練、微調並交付模型。AI 工程師則把既有模型（通常透過 API）組合成產品。ML 工程師把時間花在資料集與訓練迴圈上。AI 工程師則把時間花在提示、RAG、代理、評估與延遲上。在同時做這兩件事的大型公司，這條界線會變得模糊。請參閱 [轉職指南](../TRANSITION_GUIDE.md)。

### 我要如何成為 AI 工程師？

如果你已經會寫生產程式碼，落差其實不大：學習 LLM 的行為模式、學習 RAG 與代理模式、學習評估，再學會一套推論技術棧即可。[轉職指南](../TRANSITION_GUIDE.md) 會把你目前的角色（後端、前端、QA、PM、資料、DevOps）對應到適合的 AI 角色，以及需要補齊的具體技能。

### 做 AI 工程該學哪一種程式語言？

Python 是 AI 工作的預設語言。TypeScript 是最常見的第二語言，因為前端與邊緣代理技術棧都在這裡。C# 與 Go 則出現在企業基礎設施的角色中。大部分生產 AI 程式碼讀起來就像一般的應用程式碼：HTTP 客戶端、佇列消費者、資料庫呼叫，再加上對某個模型供應商的呼叫。

### AI 工程是一份好的職涯選擇嗎？

需求強勁，薪資對標資深軟體工程師，在前沿實驗室往往更高。風險在於技術棧變動很快：去年還很重要的框架，今天可能已經被淘汰。能在每次版本更迭中持續累積的技能，是評估、系統設計與有接地的除錯。請參閱 [就業市場趨勢](06-job-market-trends-2026.md)。

## RAG

### 什麼是 RAG？

檢索增強生成（Retrieval-Augmented Generation）是一種模式：在查詢當下取得外部上下文（文件、資料列、程式碼、影像），放進 LLM 的提示裡，讓模型能以此接地作答，而不是從訓練資料中產生幻覺。對於任何需要知道訓練截止日之外資訊的 LLM 而言，這是最常見的生產模式。請參閱 [RAG 基礎](../06-retrieval-systems/01-rag-fundamentals.md)。

### RAG 是如何運作的？

使用者查詢會被轉換成一個搜尋請求，系統從知識儲存（向量資料庫、關鍵字索引、圖，或三者的混合）中檢索出最相關的區塊，將它們重排序，再把排名最前的結果作為上下文傳給 LLM 進行生成。兩個失敗點分別是檢索（正確的區塊沒有被取出）與生成（模型忽略或誤用了該區塊）。大多數 RAG 失敗都是檢索失敗。

### 既然有了長上下文視窗，RAG 是不是已經沒用了？

並沒有。即使在 Claude Opus 4.7、Claude Sonnet 4.6、GPT-5.5 與 Gemini 3.1 Pro 上有了 1M 到 2M token 的上下文視窗，RAG 仍在成本、延遲、資料新鮮度與語料規模上勝出。企業資料集（SharePoint、日誌封存、程式碼 monorepo）會超出任何上下文視窗。RAG 扮演的是過濾器，負責找出那值得放進高價值視窗的 0.01% 資料。請參閱 [RAG vs 長上下文](../06-retrieval-systems/14-production-rag-at-scale.md)。

### RAG 與微調有什麼差別？

RAG 在查詢當下透過上下文注入知識。微調則把行為烙印進權重裡。經驗法則是：**RAG 用於事實，微調用於形式**。當知識會變動、需要引用來源，或資料必須留在模型外部時，使用 RAG。當你想要一致的語氣、嚴格的輸出格式，或在重複任務上獲得更低延遲時，使用微調。請參閱 [微調策略](../03-training-and-adaptation/02-fine-tuning-strategies.md)。

### 最好的向量資料庫是哪一個？

沒有單一的最佳解。Pinecone 在受管規模與 SLA 上勝出。Qdrant 在開源速度上領先（在 1000 萬向量下大約 12ms p99）。Weaviate 擁有最強的原生混合檢索（在單一查詢中結合 BM25 + 稠密向量 + 中繼資料）。一旦你需要超過 5000 萬向量的分散式規模，Milvus 是選擇。如果你本來就在用 Postgres、而且索引在 1000 萬向量以下，pgvector 就是正確答案。請參閱 [向量資料庫](../06-retrieval-systems/04-vector-databases.md)。

### 什麼是上下文檢索（contextual retrieval）？

上下文檢索是 Anthropic 提出的技術，在對每個區塊做嵌入與索引之前，先在區塊前面附上一段由 LLM 生成的簡短上下文摘要，讓區塊帶著它在文件中的位置一起被檢索。Anthropic 回報，搭配混合搜尋時檢索失敗減少 49%，再結合重排序器時則達到 67%。請參閱 [上下文檢索](../06-retrieval-systems/10-contextual-retrieval.md)。

### 什麼是混合搜尋（hybrid search）？

混合搜尋結合稀疏的關鍵字檢索（通常是 BM25）與稠密的向量檢索，並把兩份排名清單融合成一份，通常使用 Reciprocal Rank Fusion。稀疏這一側負責捕捉精確的 token（產品代碼、函式名稱、罕見名詞）；稠密這一側負責捕捉同義詞與意圖。每一個現代向量資料庫都內建即用的混合搜尋。請參閱 [混合搜尋](../06-retrieval-systems/05-hybrid-search.md)。

### 什麼是 GraphRAG？

GraphRAG 會從語料中抽取實體與關係，建立一個知識圖，並以圖的走訪（而非向量相似度，或與其並用）來查詢。對於**彙整型問題**（「彙整這 50 份合約中所有的法律風險」）這種向量 RAG 只會回傳相關但彼此不連貫區塊的情境，它是正確的模式。Microsoft 的 LazyGraphRAG 把昂貴的社群摘要延後到查詢當下才做，藉此降低匯入成本。請參閱 [GraphRAG](../06-retrieval-systems/07-graph-rag.md)。

### RAG 最佳的分塊大小是多少？

沒有放諸四海皆準的答案。對於散文，300 到 500 token 的區塊搭配 50 token 重疊是合理的預設值。程式碼與結構化資料則需要更大的區塊（1000 到 2000 token）。更大的收益來自**結構感知分塊**（在標題、段落、程式碼區塊處切分）、**上下文分塊**（在前面附上摘要），以及**階層式分塊**（索引小區塊，回傳其父層上下文）。請參閱 [分塊策略](../06-retrieval-systems/02-chunking-strategies.md)。

## Agents

### 什麼是 AI 代理？

AI 代理是一種系統，由 LLM 決定下一步要做什麼、執行一個工具、觀察結果，然後再次決定，全部在一個迴圈中進行。最簡單的代理是 ReAct 模式：思考（Thought）→ 行動（Action）→ 觀察（Observation）→ 重複。現代代理（Claude Opus 4.7、GPT-5.5 reasoning、DeepSeek-R2）把推理步驟直接烙進模型本身。請參閱 [代理基礎](../07-agentic-systems/01-agent-fundamentals.md)。

### 代理與聊天機器人有什麼差別？

聊天機器人回應一則訊息。代理則在世界中**採取行動**：執行程式碼、呼叫 API、讀取檔案、傳送訊息、預約行程。這個區別之所以重要，是因為行動很難回復，所以代理需要非常不同的防護機制、沙箱化，以及人在迴路中的模式。請參閱 [代理式安全](../07-agentic-systems/09-agentic-security-and-sandboxing.md)。

### 什麼是 MCP（Model Context Protocol）？

MCP 是一個開放協定，讓 LLM 應用程式能透過標準介面連接到工具與資料來源。Anthropic 於 2024 年 11 月推出它。治理權於 2025 年 12 月移交給 Linux Foundation 的 Agentic AI Foundation。採用已遍及各方：Anthropic、OpenAI、Google、Microsoft、AWS 全都支援它。截至 2026 年 5 月，已有超過 2,300 個公開的 MCP 伺服器。請參閱 [工具使用與 MCP](../07-agentic-systems/03-tool-use-and-mcp.md)。

### 最好的代理框架是哪一個？

三大框架涵蓋了大部分的生產需求。**LangGraph**（基於圖、於 2026 年初星標數超越 CrewAI）是有狀態多代理控制流程搭配檢查點時的預設選擇。**CrewAI**（現為 v1.13，被超過 60% 的財星 500 大企業使用）是以角色為基礎的商業自動化的選擇。**Microsoft Agent Framework**（2026 年 2 月推出 RC 1.0，2026 年第二季 GA）是整合 AutoGen 與 Semantic Kernel 之後、面向企業 .NET 與 Python 的後繼者。請參閱 [框架選型指南](../09-frameworks-and-tools/08-framework-selection-guide.md)。

### 什麼是代理式 RAG（agentic RAG）？

代理式 RAG 把線性的「先檢索再生成」管線，替換成一個迴圈：由代理決定要檢索什麼、評估結果是否夠好，若不夠好就重新查詢。相關模式包括 Self-RAG（模型發出反思 token）、Corrective RAG（獨立的評分器）、Adaptive RAG（由分類器挑選檢索深度），以及多跳分解。請為每次查詢預留 8 到 12 秒、3 到 4 次迭代的預算。請參閱 [代理式 RAG](../06-retrieval-systems/08-agentic-rag.md)。

### 電腦使用（computer-use）代理是如何運作的？

電腦使用代理會對桌面或瀏覽器擷取畫面截圖，決定一個滑鼠與鍵盤動作，執行它，然後再擷取另一張截圖。三個生產選項分別是 Claude Computer Use（透過截圖、可跨作業系統移植）、Google Gemini Computer Use（透過 DOM 感知、針對瀏覽器最佳化），以及 OpenAI Operator（聚焦於網頁任務）。Claude Sonnet 4.6 在 OSWorld-Verified 上達到 72.5%，相較 2024 年 10 月推出時的 14.9% 大幅提升。請參閱 [電腦使用代理](../17-tool-use-and-computer-agents/04-computer-use-agents.md)。

### 什麼是上下文工程（context engineering）？

上下文工程是策劃代理在每一個推論回合中所看到的完整 token 集合（系統提示、工具、檢索到的資料、先前的工具結果、訊息歷史），相對於提示工程只寫一次好的指令。它之所以重要，是因為長時間執行的代理會出現**上下文腐化（context rot）**：當視窗被陳舊的工具輸出填滿時，準確度會下降。核心技巧有壓實（summarize 並重啟迴圈）、即時載入（保留參照、按需取得）、結構化筆記（把進度寫在視窗之外），以及子代理隔離（把細節繁重的子任務委派給一個乾淨的視窗，由它回傳一段簡短摘要）。目標是每回合都維持最小、最高訊號量的 token 集合。請參閱 [上下文工程](../05-prompting-and-context/05-context-engineering.md)。

### 什麼是 Agent Skills？

Agent Skills 是一組由指令、腳本與資源組成的資料夾，代理會按需載入，藉此在某項任務上專精。每個技能都是一個帶有 YAML 中繼資料的 `SKILL.md` 檔案，外加可選的隨附檔案。它們採用**漸進式揭露（progressive disclosure）**：名稱與描述放在系統提示中，完整的 `SKILL.md` 只在代理判斷相關時才載入，被參照的檔案也只在需要時才載入，藉此保持上下文精簡。Skills 與 MCP 互補：MCP 把代理連接到工具與資料，Skills 則教會它使用這些工具的工作流程。請參閱 [打造工具使用代理](../17-tool-use-and-computer-agents/05-building-tool-agents.md)。

## Models

### 現在最好的 LLM 是哪一個？

在 2026 年 6 月並沒有單一的最佳模型，但能力天花板已被推高：**Claude Fable 5**（6 月 9 日）是 Anthropic 迄今最有能力、且廣泛釋出的模型，一個帶有安全防護的 Mythos 級模型，定價為每 1M $10/$50。在天花板之下，排行榜因任務而分裂。**Claude Opus 4.8** 以 69.2% 在 SWE-Bench Pro 上領先長時程編碼，價格僅為 Fable 的一半。**GPT-5.5** 守住 SWE-bench Verified（88.7%）與 Terminal-Bench。**Gemini 3.1 Pro** 在 GPQA Diamond 上以 94.3% 領先科學推理。**Claude Sonnet 4.6** 以 $3/$15 維持其性價比主力的地位。請參閱 [模型分類](../02-model-landscape/01-model-taxonomy.md)。

### Claude / GPT / Gemini / DeepSeek 的價格是多少？

定價每個月都在變。截至 2026 年 6 月，前沿級的封閉模型大約是每百萬輸入 token $3 到 15、每百萬輸出 token $15 到 75，而快取在重複的前綴上能把這數字砍掉 75% 到 90%。中階模型（Claude Sonnet 4.6、GPT-5.5-mini、Gemini 3.1 Flash）大約是每百萬 $0.30 到 3 / $1 到 15。**DeepSeek 重設了地板價**：V4 Pro 為每 1M $0.435 / $0.87（75% 折扣於 2026 年 5 月 22 日成為永久），而 V4 Flash 為每 1M $0.14 / $0.28 並搭配 1M 上下文視窗，在許多任務上大約比封閉前沿便宜 10 倍。請務必對照供應商的定價頁面以取得當前費率。請參閱 [定價與成本](../02-model-landscape/03-pricing-and-costs.md)。

### Claude Opus 與 Claude Sonnet 有什麼差別？

Opus 是 Anthropic 的前沿級（Opus 4.8：4.x 系列中最聰明的，$5/$25）。Sonnet 是生產主力：大約有 Opus 90% 的品質，價格卻是其 60%。Haiku 是快速級：便宜、低延遲，適合做路由與分類。在這三者之上是 Claude Fable 5（$10/$50），是值得付出 2 倍 Opus 價格的工作所對應的能力天花板。正確的模式是把容易的查詢路由到 Haiku、中等的路由到 Sonnet、困難的路由到 Opus，只有受天花板限制的工作才交給 Fable。請參閱 [模型選型](../02-model-landscape/04-model-selection-guide.md)。

### 我該使用開源模型嗎？

在高流量下的每次查詢成本、資料落地需求，或微調需求上，答案是肯定的。開放權重的品質已經拉近到封閉前沿的 5 到 15 分之內（Qwen3-Embedding-8B 在 MTEB Multilingual 上領先，Llama 4 Maverick 與 DeepSeek V4 Pro 都是具競爭力的前沿選擇）。代價在於營運面：你得自己擁有推論技術棧、GPU 帳單，以及安全修補。請參閱 [模型概覽](../02-model-landscape/01-model-taxonomy.md)。

### 什麼是提示快取（prompt caching）？

提示快取會把固定提示前綴的 KV cache 保持在推論伺服器上的熱狀態，讓後續呼叫只需為新的 token 付費。所有主要供應商都支援它。快取讀取比全新 token 便宜 75% 到 90%。但要注意：快取寫入通常會多花 25%，所以只有在 TTL（通常為 5 分鐘）內至少重複使用同一前綴 3 到 5 次時，快取才划算。請參閱 [KV Cache 與上下文快取](../04-inference-optimization/02-kv-cache-and-context-caching.md)。

## Evaluation

### 你要如何評估一個 LLM？

LLM 評估是分層的：**無參照指標**（透過 LLM-as-judge 評估忠實度、相關性、連貫性）用於快速迭代、**黃金測試集**用於回歸測試，以及**任務特定指標**（QA 用精確匹配、翻譯用 BLEU、程式碼用 pass@k）。針對 RAG，特別要使用 RAG 三元組：上下文相關性、忠實度、答案相關性。請參閱 [LLM 評估](../14-evaluation-and-observability/01-llm-evaluation.md)。

### 什麼是 LLM-as-judge？

LLM-as-judge 用一個 LLM 依據評分標準（正確性、有用性、安全性）為另一個 LLM 的輸出評分。它能擴展到人工評估無法企及的規模，但有已知的偏誤：位置偏誤、冗長偏誤、自我偏好偏誤。標準做法是使用一個更強的模型當評審（Claude Opus 4.8 或 GPT-5.5 reasoning）、隨機化位置，並對照一小份人工標註的樣本來驗證。

### 最好的 LLM 可觀測性工具是哪一個？

領先的平台有 **Langfuse**（最佳的自架開源方案，於 2026 年 1 月被 ClickHouse 收購）、**Braintrust**（最適合搭配品質閘門做評估驅動的 CI/CD）、**LangWatch**（最適合代理模擬）、**LangSmith**（LangChain 原生），以及 **Arize Phoenix**（OTel 原生）。請依部署模式（SaaS vs 自架）、是否需要 CI/CD 閘控，以及你對特定框架的依賴程度來選擇。請參閱 [可觀測性](../14-evaluation-and-observability/02-observability.md)。

### 什麼是 RAGAS？

RAGAS（Retrieval Augmented Generation Assessment）是一個用於 RAG 評估的 Python 函式庫。它提供無參照指標（忠實度、答案相關性、上下文相關性）與基於參照的指標（上下文召回率、上下文精確率），全部透過 LLM-as-judge 計算。它是任何 RAG 評估管線實質上的起點。請參閱 [RAG 評估](../06-retrieval-systems/13-rag-evaluation-patterns.md)。

### 你要如何在生產環境中偵測並處理模型漂移？

漂移來自兩個面向：**供應商**悄悄更新了模型（或你遷移了版本），以及你的**流量**偏離了當初調校提示與評估時所依據的分布。偵測方式：明確固定模型版本、每天針對固定版本與最新版本執行一套金絲雀評估、追蹤輸出分布的統計量（長度、拒答率、格式有效性）與在抽樣即時切片上由評審評分的品質，並對差異發出警報。回應方式：一份凍結的黃金集能告訴你究竟是模型變了還是你的流量變了；在採用新版本之前先重跑提示評估，並讓前一版本保持熱狀態以便立即回滾。把無法解釋的品質下滑當成一起需要 on-call 路徑的事件來處理，而不是一件好奇之事。

## Inference

### 什麼是 vLLM？

vLLM 是一個開源的 LLM 推論引擎，首創了 PagedAttention（以虛擬記憶體式的方式配置 KV cache）。當工作負載是「在連續批次處理下跑 Llama、Mistral、Qwen 或 DeepSeek」時，它是預設的開源推論引擎。在主要引擎中，它最容易營運、修補也最完善。由於 2026 年 2 月的一個 CVE，多模態部署必須執行 v0.18.2 以上版本。請參閱 [服務基礎設施](../04-inference-optimization/06-serving-infrastructure.md)。

### vLLM 與 SGLang 有什麼差別？

兩者都是開源推論引擎。vLLM 有更廣的模型涵蓋度與營運成熟度。SGLang 在結構化輸出與函式呼叫的工作負載上，吞吐量大約高出 29%，這要歸功於非同步約束解碼，以及透過 RadixAttention 達成的同級最佳前綴快取重用。重要警告：截至 2026 年 5 月，SGLang 的多模態路徑有未修補的 CVE，所以多模態流量應改在 vLLM v0.18.2 以上版本執行。

### 什麼是 TensorRT-LLM？

NVIDIA 的推論引擎。在 H100/H200/B200 上提供比 vLLM 與 TGI 高 2 到 4 倍的吞吐量，但代價是 1 到 2 週的建置時間，以及對 NVIDIA 的硬性鎖定。當你已承諾投入 NVIDIA 算力、而且吞吐量的差距足以抵償那份營運稅時，它就是正確選擇。

### 你要如何最佳化 LLM 推論成本？

五個高槓桿的做法：**模型串接**（把容易的查詢路由到小模型、困難的路由到前沿）、**提示快取**（重複前綴省下 75% 到 90%）、**語意快取**（對相似查詢完全略過 LLM 呼叫）、**量化**（用 FP8 或 4-bit 權重在一顆 GPU 上塞進更多模型），以及**連續批次處理**（vLLM/SGLang 在迭代層級做批次）。把它們組合起來，經常能在不損失品質的前提下把推論成本砍掉 10 倍。請參閱 [成本最佳化](../04-inference-optimization/07-cost-optimization-playbook.md)。

### 什麼是推測解碼（speculative decoding）？

推測解碼讓 LLM 在每一次前向傳遞中生成多個 token：先由一個較便宜的「草稿」模型（或主模型上的額外頭，如 Medusa）預測接下來的幾個 token，再於目標模型上以單次平行傳遞一次驗證它們全部。其收益是 2 到 3 倍更快的實際生成時間，且零品質損失。已內建於 vLLM 與 TensorRT-LLM。請參閱 [推測解碼](../04-inference-optimization/03-speculative-decoding.md)。

### 什麼是 token 預算，你要如何落實它？

token 預算是在所選範圍上對 token 消耗設下的硬性上限：每個請求（最大輸入加上 `max_tokens` 輸出）、每個使用者或租戶每天、每個代理任務（步數預算，讓失控的迴圈無法燒掉整月的支出），以及昂貴層級下每個團隊每月。落實的位置在閘道，而不在提示裡：在派送前先計算 token、拒絕或降級超過單一請求上限的請求、以原子方式遞減租戶配額，並以乾淨的升級流程終止超出步數預算的代理執行。實務上的搭配是預算加上路由：當某個租戶接近配額時，路由器會把它降級到較便宜的層級，而不是直接切斷。

### 你要如何跨多個 LLM 供應商設計備援？

把供應商當成既有政策差異、也有可用性差異的不可靠依賴，而不只是上線時間的差異。模式為：每個任務類型一個主要供應商搭配一到兩個備援、針對每個供應商以錯誤率、p95 延遲與速率限制餘裕設置斷路器，並在觸及硬性限制之前先做預先的流量轉移。三個細節把生產級設計與白板上的設計區分開來：**與供應商配對的提示**（為 Claude 調校的提示在 GPT-5.5 上會表現不佳，所以提示變體要隨供應商一起版本控制）、**快取經濟性**（故障切換會重設你的前綴快取，所以一個保持熱狀態的主要供應商可能勝過名義上更便宜的冷備援），以及**政策感知的備援**（某個供應商可能拒絕你產品所需的內容類別，這是你的健康檢查無法捕捉的一類故障）。請每月透過一次演習日演練來驗證整條鏈。

## Memory

### 最好的 AI 代理記憶框架是哪一個？

截至 2026 年 5 月，四個成熟的選項是 **Mem0**（最廣泛的獨立記憶層，在 LoCoMo 上得 92.5、在 LongMemEval 上得 94.4）、**Zep**（具時間感知的生產管線，內建原生對話摘要）、**Letta**（為長時間執行、具無界記憶的代理提供作業系統式分頁），以及 **Cognee**（以知識圖為先，面向 RAG 密集的工作流程）。請依使用情境選擇：聊天機器人個人化 → Mem0；規模化的生產代理 → Zep；長時程任務代理 → Letta；以知識圖接地的 RAG → Cognee。請參閱 [代理式記憶](../08-memory-and-state/04-agentic-memory-mem0.md)。

### 代理中的短期記憶與長期記憶有什麼差別？

短期記憶存在於 LLM 的上下文視窗中（當前回合、工具輸出、暫存區）。長期記憶則跨工作階段持久化在向量資料庫、圖，或關聯式儲存中。記憶架構還會進一步細分：**情節式**（過往的軌跡）、**語意式**（從使用者或世界中抽取出的事實）、**程序式**（學到的技能、操作手冊）。為某項事實挑選正確的層級很重要：一個被提升到長期記憶的工作階段偏好，會跨工作階段外洩。請參閱 [記憶架構](../08-memory-and-state/01-memory-architectures.md)。

### 知識圖如何幫助 AI 代理？

知識圖會明確地儲存實體與關係（User → OWNER_OF → Project_A）。它賦予代理對結構化關係的**確定性**檢索，這是向量搜尋做不到的。最強的模式是混合式：向量搜尋以相似度找到入口節點，圖的走訪再展開相關的上下文。用於法遵、多跳推理，以及任何關係屬於一等公民的領域（法律、生醫、金融）。請參閱 [長期記憶](../08-memory-and-state/03-long-term-memory.md)。

## Security

### 什麼是提示注入（prompt injection）？

提示注入是 SQL 注入的 LLM 時代版本：使用者輸入或被檢索文件中的惡意內容覆蓋了系統指令，讓模型去做它不該做的事。**直接注入**位於使用者提示中。**間接注入**則藏在模型會讀取的文件裡（一個網頁、一封電子郵件、一份 PDF）。OWASP LLM Top 10 把它列為 LLM 的第一大風險。請參閱 [提示注入防禦](../05-prompting-and-context/08-prompt-injection-defense.md)。

### 你要如何防範提示注入？

沒有萬靈丹：提示注入無法像 SQL 那樣被完全「跳脫」。生產級的防禦堆疊會結合：**輸入隔離**（用 XML 標籤標記不受信任的內容）、**雙 LLM 模式**（在主模型看到輸入之前，先由一個小型守衛模型分類其意圖）、**金絲雀 token**（偵測模型是否外洩了它的系統提示）、**最小權限工具範圍**，以及對破壞性工具呼叫的**人在迴路中**審核。請參閱 [代理式安全](../07-agentic-systems/09-agentic-security-and-sandboxing.md)。

### 什麼是 OWASP LLM Top 10？

OWASP Top 10 for LLM Applications（v2.0，於 2025 年發布）是 LLM 安全風險的權威清單。最前面的項目包括：提示注入、不安全的輸出處理、訓練資料投毒、模型阻斷服務、供應鏈漏洞、敏感資訊外洩、不安全的外掛設計、過度代理權、過度依賴，以及模型竊取。針對代理式應用的 2026 年更新，把目標劫持、身分濫用與連鎖故障加入為最高風險。請參閱 [LLM 安全](../12-security-and-access/01-llm-security.md)。

### AI 代理中的沙箱化是什麼？

沙箱化把代理所生成並執行的程式碼與主機系統隔離開來。標準模式使用短暫的微型虛擬機（E2B、Docker、Firecracker），它們在 10ms 內啟動、執行程式碼、然後被銷毀。沒有沙箱化，一個被提示注入的代理就能 `rm -rf /` 或外洩機密。有了沙箱化，最壞的情況也只是一個被銷毀的拋棄式容器。
