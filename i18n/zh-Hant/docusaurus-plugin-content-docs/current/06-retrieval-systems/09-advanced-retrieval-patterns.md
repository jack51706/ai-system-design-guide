# 進階檢索模式

在基礎之上，生產環境的 RAG 系統會運用專門的模式來處理複雜的查詢與文件落差。這些模式是高精度搜尋的「獨門秘方」，而且越來越常被整合進託管式 RAG 服務中。

## 查詢分解（Multi-Query）

複雜的使用者查詢往往屬於「複合查詢」。
- **使用者**：「比較我們第三季與第四季的營收，並解釋為何下滑。」
- **分解**：
  1. 「第三季營收」
  2. 「第四季營收」
  3. 「第四季營收變動的原因」
- **實作方式**：用 LLM 產生這 3 個子查詢，針對全部子查詢搜尋資料庫，再彙整出上下文。

## 假設性文件嵌入（HyDE）

查詢很短，文件很長。這種「不對稱性」會造成檢索失敗。
- **模式**： 
  1. 取得使用者查詢。
  2. 詢問 LLM：「為這個問題寫出一段假設性的答案。」
  3. **改為嵌入這段假設性答案**，而不是嵌入查詢本身。
- **為什麼？**：假設性答案與真實文件位於相同的「向量鄰域」，因此能帶來高得多的召回率。

## 情境式檢索（Anthropic 模式）

此模式在 2024 年底由 Anthropic 標準化，用來解決**情境稀釋（Context Dilution）**的問題。

- **問題所在**：某個區塊可能寫著「它要價 $200」，但少了標題，我們就不知道「它」指的是「Widget-X」。
- **模式**：在資料攝入期間，針對每一個 300 個 token 的區塊，讓 LLM 撰寫一段 50 個 token 的情境字串（例如「本區塊談的是 Widget-X 在北美市場的定價」）。
- **效益**：對於零散的資料，可將檢索精度提升 30 至 50%。

## 迭代式文件擴充

我們不再只是儲存原始文件，而是改為儲存「擴充過」的中介資料（meta-data）。
- **摘要**：儲存一段文件的摘要。
- **問答生成**：產生這份文件能回答的 5 個問題，並將這些問題*連同*文件一起嵌入。
- **現況**：多數高階 RAG 系統現在嵌入的是**「問題」**而非**「答案」**，以便對應使用者的查詢意圖。

## 情境內重排序

如今 1M 至 2M 的上下文視窗已是標準配備（Claude Sonnet 4.6、Gemini 3.1 Pro），因此**依情境排序（Rank-by-Context）**成為一種可行的模式。
1. 檢索出前 100 份文件。
2. 把全部 100 份放進上下文視窗。
3. 詢問模型：「閱讀這 100 份文件，找出最相關的 5 份。接著用這 5 份來作答。」
- **優勝之處**：這種做法善用了模型的**長上下文推理（Long Context Reasoning）**能力來執行重排序，而無需另外使用一個 Cross-Encoder 模型。

## 面試問題

### Q：為什麼 HyDE（假設性文件嵌入）對某些應用而言有風險？

**有力的回答：**
HyDE 仰賴「幻想」出一個基準答案來尋找真實資料。如果使用者的查詢描述的是不存在或邏輯上不可能的事物，LLM 仍然會產生一個假設性答案。這可能從資料庫中拉進「錯誤但語意相似」的資料，進而強化模型一開始的幻覺。標準的緩解做法是採用**混合式做法**：用真實查詢（關鍵字）檢索一次，再用 HyDE 查詢檢索一次，接著用 **RRF** 把兩者結合起來。

### Q：什麼是「不對稱檢索（Asymmetric Retrieval）」問題？

**有力的回答：**
不對稱檢索指的是這個事實：使用者查詢通常很短（3 至 10 個字），而文件區塊很長（300 至 500 個字）。兩者在向量空間中分屬不同的統計分布，因而導致「距離偏差（Distance Bias）」。高效能系統會用**不對稱編碼器（Asymmetric Encoders）**（一個模型處理查詢，另一個處理文件）或**查詢擴展**（HyDE）來解決，把查詢「膨脹」成類似文件的分布。
