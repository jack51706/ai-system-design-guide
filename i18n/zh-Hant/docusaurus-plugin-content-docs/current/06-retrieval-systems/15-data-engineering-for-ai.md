# AI 的資料工程

模型的好壞，取決於餵給它的資料，而任何認真的 AI 系統中，大部分工作都發生在模型的上游。本章涵蓋同時餵養 RAG 與微調的**資料層**：擷取、清理、去重、PII 與治理、品質過濾，以及把這一切搬運起來的管線。它被放在檢索章節，是因為大多數團隊最先在這裡碰到它，但它其實是**橫切**的：同一條把文件準備給向量資料庫的管線，也會把範例準備給微調使用。把資料工程當成「RAG 的第二步」是低估了它，它是一門獨立的學科，上述兩種消費者都依賴它。

## 共用管線

核心理念是：一條管線，兩種消費者。RAG 需要文件經過解析、清理、去重、分塊、豐富化與嵌入。微調需要範例經過解析、清理、去重、品質過濾、對評估集做去汙染、平衡，再格式化。前五個階段是**共用**的；兩者只在尾端分流。

```
SOURCES        web · docs (PDF/office/email) · DBs/APIs · uploads
   |
[1] DETECT & ROUTE    MIME sniff -> language ID -> route by type
[2] PARSE / EXTRACT   typed elements + structure
[3] CLEAN / NORMALIZE boilerplate strip · encoding repair · unicode · language filter
[4] DEDUP             exact -> MinHash/LSH -> semantic (SemDeDup)
[5] GOVERN            PII detect/redact · license/consent/provenance metadata
[6] QUALITY FILTER    heuristics + model classifier  [always ablate on a downstream eval]
   |-- RAG branch ---------------------|   |-- FINE-TUNE branch ----------------|
   | [7a] chunk -> enrich -> embed     |   | [7b] curate/label -> synthesize    |
   |      -> VECTOR STORE (CDC-fresh)  |   |      -> decontaminate -> balance   |
   |-----------------------------------|   |      -> TRAINING RECORDS           |
CROSS-CUTTING   orchestration (Airflow/Dagster) · lineage (OpenLineage) · versioning (DVC/lakeFS)
```

「大部分工作」這個說法是站得住腳的：廣為引用的經驗法則說實務工作者花費約 80% 的時間在資料準備上，這屬於民間傳說等級（在許多來源間被反覆轉述，而非實際量測得來），但方向是對的；而供應商宣稱 AI 代理現在能把準備時間縮減 50-70%，這既未經查證又帶有自利動機。兩者都當成方向性參考即可。

## 擷取

輸入是一堆異質檔案（PDF、掃描檔、office 文件、HTML、email、圖片）。你必須偵測每個檔案是什麼，把它導向正確的解析器，並產出一份正規化的結構化表示。

- **依內容而非副檔名來偵測檔案類型。** 標準做法是透過 `libmagic`（Unix `file` 指令背後的函式庫）做魔術數字嗅探，並透過像 `python-magic` 這樣的綁定來使用。Unstructured 這類解析器就是以這種方式自動偵測類型，並在失敗時退回到副檔名。
- **語言偵測**用於路由與過濾，常見做法是使用 fastText 的語言分類器。
- **解析與路由。** 內容類型決定處理器：可抽取文字的 PDF 走快速文字路徑，掃描或圖片型 PDF 走 OCR，複雜的多欄或表格密集文件走版面模型或多模態（逐頁截圖）路徑，office 文件走特定格式的處理器，email 則拆分成標頭與內文。領先的工具包有 **Unstructured**（支援 64 種以上檔案類型，產出像 Title/NarrativeText/Table 這類具型別的元素）、**Docling**（IBM 出品，採 MIT 授權，具備版面與表格模型，支援多種匯出格式），以及 **LlamaParse**（分層的 API 模式，最高可達多模態代理式解析）。一個教學重點：已發表的解析器基準測試彼此相左，所以**請拿你自己的文件來做解析器選型的基準測試**，而不是信任排行榜。

## 清理與正規化

共用的第一道品質關卡：
- **樣板移除。** 移除導覽列、廣告、頁首、頁尾、cookie 橫幅。針對大規模 HTML，標準做法是使用內容抽取函式庫；FineWeb 專案發現，用這類工具從原始網頁封存檔中抽取，勝過使用預先抽取好的文字（後者「保留了過多樣板」），所以這屬於上游品質問題，而非表面修飾。
- **編碼修復**，處理亂碼（mojibake）與雙重編碼的文字，外加換行符正規化。
- **Unicode 正規化**（NFC/NFKC），讓視覺上相同的字串能比較為相等，這是去重與比對能運作的前提。
- **語言過濾**，剔除低於信心門檻的內容。

一個值得教的反直覺警語：**過濾並非越多越好。** FineWeb 的消融實驗發現許多啟發式規則的影響力低微或邊際（它因為效益太小而捨棄了大多數候選過濾器），而 Nemotron-CC（arXiv:2412.02595）回報，激進的啟發式過濾可能丟棄約 18% 被它自家品質分類器評為高品質的 token。過濾器必須針對下游評估做消融，而不是憑空假設其效果。

## 去重

去重是槓桿最高的階段，也是「共用基礎設施」最清楚的範例，因為它能從三個面向帶來回報。奠基性的成果（Lee et al., arXiv:2107.06499）回報，對訓練資料去重能讓模型輸出記憶下來的文字的頻率降低約 10 倍，以更少的步數達到相等或更佳的準確率，而且關鍵在於，**它能減少訓練集與測試集的重疊，所以去重同時也是去汙染。** 它也藉由減少對重複 PII 的記憶來緩解隱私風險。

這三個層級以串接方式執行（先做便宜且精確的，最後做昂貴且語意的）：
1. **精確**：對整份文件或正規化後的子字串做雜湊。快速且精準；但只能抓到逐字的複本。
2. **模糊／近似重複**：**MinHash + LSH** 在 token n-gram shingle 上估計 Jaccard 相似度，並把候選者分桶以避免兩兩比對（這是網路規模的預設做法；現在有些向量資料庫已將它作為原生索引內建）。SimHash 是經典的替代方案。警語是：這些方法是*詞彙層級*的，所以共用某個範本但語意不同的文件可能被誤刪。
3. **語意／嵌入去重**：**SemDeDup**（arXiv:2303.09540）對每個項目做嵌入、分群，並在群集內依餘弦相似度丟棄近似重複者；它回報在一個大型圖文資料集上移除了約 50% 的內容，而效能損失極小，能抓到 MinHash 漏掉的改寫。

生產環境的模式會把它們組合起來，先 MinHash，再 SemDeDup。把這三面向的回報明確講出來：RAG（重複的區塊浪費上下文視窗，並排擠掉多樣的證據）、訓練（更少的記憶、更少的步數），以及評估完整性（對基準測試去重可避免汙染）。

## PII、同意與治理

**PII 偵測與遮蔽。** Microsoft Presidio（MIT 授權）是開源標準：一個透過 NER 加上正規表示式、檢核碼與上下文詞彙來偵測實體的 Analyzer，以及一個能對文字、圖片（搭配 OCR）與結構化資料進行遮蔽、替換、遮罩、雜湊或加密的 Anonymizer，可在語料庫規模上部署。由於去重能減少重複所驅動的記憶，隱私與去重是連動的。

**同意、授權與來源出處**如今是法規要求，而不只是衛生習慣。在 EU AI Act 下，通用模型供應商必須維護一份著作權政策，並使用 AI Office 的強制範本發布一份訓練內容的「足夠詳盡的摘要」，內容包含資料來源以及對著作權退出選項的尊重（這項摘要義務自 2025 年 8 月起適用於新的通用模型，已上市的模型則寬限至 2027 年 8 月）。輸出標記義務隨之而來。對管線的治理意涵是：每一筆記錄從擷取開始就要把來源、授權、同意狀態與時間戳記當成中繼資料隨身攜帶，而這正是血緣（見下文）所提供的。治理不是最後一道關卡；它是貫穿每一個階段的中繼資料。參見 [AI Governance and Compliance](../13-reliability-and-safety/04-ai-governance-and-compliance.md)。

## 品質過濾與豐富化

**品質過濾**分成兩大類。**啟發式**規則（長度、符號對詞彙比、重複度、停用詞出現情況）便宜，但無法抓到複雜的內容雜訊。**模型式分類器**為品質或教育價值評分；FineWeb-Edu 在 LLM 生成的品質標註上訓練了一個輕量分類器，並回報了可觀的下游增益，能以遠少於原本的 token 量匹配上一個更大的語料庫。需要標註的警語是：分類器過濾不是免費的午餐（「資料品質錯覺」的研究主張它可能校準失準），所以**永遠要在下游評估上對過濾器做消融，絕不要憑名聲信任它們。**

**分塊**（RAG 端的階段）沒有放諸四海皆準的贏家：已發表的基準測試在固定大小、遞迴與語意策略之間擺盪幅度很大，所以它是因資料集而異的，必須經過評估，不能用預設值帶過。參見 [Chunking Strategies](02-chunking-strategies.md)。

**豐富化與中繼資料。** 附上標準中繼資料（標題、作者、時間戳記、來源、章節）與生成的中繼資料（區塊摘要、上下文前綴、某個區塊能回答的合成問題），這能把單一軸向的向量相似度變成多維度的過濾式搜尋。參見 [Contextual Retrieval](10-contextual-retrieval.md)。

**血緣。** OpenLineage（以 Marquez 作為參考實作）是追蹤執行、作業與資料集事件的廠商中立標準；對 ML 而言，它把這張圖往前延伸，貫穿特徵表、模型與預測。搭配資料版本控制（DVC、lakeFS、Delta/Iceberg 的時光回溯），這就是讓治理中繼資料能端到端可稽核的底層基質。

## 管線與編排

**批次 vs 串流。** 預訓練語料的準備與大量 RAG 索引建立屬於批次作業（在物件儲存上用 Spark 或 Ray）。RAG 的*新鮮度*是增量式的：當來源文件變動時，只重新擷取差異部分。Change Data Capture 即時捕捉資料列層級的來源變動，對 RAG 而言這對應到「偵測變動、新增與刪除的文件，只重新解析並重新嵌入那些文件，然後在向量資料庫中 upsert 或刪除」，避免整體重建索引，並防止陳舊或孤立的向量。

**編排器。** Airflow 是預設選項，擁有最龐大的生態系；Dagster 以資產為基礎，具備適合增量式 RAG 重新嵌入的增量重算；Prefect 則是較輕量的 Pythonic 選項。Spark 與 Ray 是編排器所排程的分散式運算，而非編排器本身。

**向量 vs 特徵儲存。** 管線在末端分岔：RAG 把區塊嵌入加上中繼資料寫入**向量資料庫**（檢索的服務層），而微調則把精選範例寫入具備時間點正確性的**特徵或訓練記錄儲存**。兩者都掛在同一條共用主幹上。

## 微調用資料

從 RAG 分流出去的尾端：
- **精選勝過數量。** 一小組精心精選的範例，往往勝過一大堆平庸的範例；品質三要素是難度、品質與多樣性。（確切的「1,000 勝過 10,000」這類數字只是示意，並非定律。）
- **合成資料**（Self-Instruct 式的做法，從一小組種子集透過教師模型擴展，再加以過濾）能便宜地擴展，但若生成時不夠謹慎，會有多樣性崩塌與模型劣化的風險，所以多樣性是頭等目標。參見 [Synthetic Data Generation](../03-training-and-adaptation/06-synthetic-data-generation.md)。
- **去汙染是非做不可的。** 需要標註的成果（arXiv:2311.04850）回報，*經過改寫或翻譯*的測試項目能溜過 n-gram 去汙染，而在這類資料上訓練的模型可以把某個基準測試過度擬合到接近前沿的分數。即使是由前沿模型生成的合成資料也被發現受到汙染。原則是：用嵌入式或 LLM 式的比對來去汙染，而不只是 n-gram 重疊，而且要在每次評估之前都做，因為汙染會悄悄地灌水分數。參見 [Benchmarks and Leaderboards](../14-evaluation-and-observability/03-benchmarks-and-leaderboards.md)。

## 失效模式

1. **信任副檔名**而非魔術位元組偵測（用錯解析器，悄悄產出垃圾）。
2. **用純文字路徑解析掃描型 PDF**（抽取結果為空或不完整；把圖片型 PDF 導向 OCR 或多模態）。
3. **從預先剝離過的 HTML 抽取**（樣板汙染；應使用內容抽取工具從原始檔抽取）。
4. **略過 Unicode 正規化**（精確去重與比對會悄悄漏掉重複項）。
5. **在錯誤的層級去重**（光靠 MinHash 會漏掉改寫；應串接精確、再模糊、再語意）。
6. **過度激進的啟發式過濾**（Nemotron-CC 回報它可能丟棄約 18% 的高品質 token；對每個過濾器做消融）。
7. **憑名聲信任品質分類器**（資料品質錯覺；應在下游驗證）。
8. **訓練前未做 PII 遮蔽**（被記憶並原樣吐出的 PII，搭配重複時情況更糟）。
9. **沒有授權、同意或來源出處的中繼資料**（無法稽核的語料庫，無法滿足訓練內容摘要的義務）。
10. **評估汙染，沉默的殺手**（只做 n-gram 去汙染會漏掉改寫過的測試項目；應使用嵌入式或 LLM 式去汙染）。
11. **陳舊的 RAG 索引**（整體重建索引而非 CDC；成本爆炸、答案陳舊、向量孤立）。
12. **預設用了不好的分塊**（基準測試彼此相左；應逐資料集評估）。
13. **沒有血緣**（無法重現的資料集與無從除錯的退步；從第一天就發出血緣事件）。

## 面試問題

### Q：為什麼去重是 AI 資料管線中最重要的階段之一？

**強答案：**
因為單一個操作能從三個不同面向帶來回報。對訓練而言，去重能削減記憶（模型原樣吐出訓練文字的情況大幅減少），並以更少的步數達到相等或更佳的準確率，因此節省運算。對 RAG 而言，重複的區塊浪費上下文視窗，並排擠掉多樣的證據，傷害檢索品質。而關鍵在於，對你的基準測試去重同時也是去汙染：奠基性的研究發現移除重複項也移除了原本在灌水評估分數的訓練集與測試集重疊。實務上我會以串接方式執行，先做精確雜湊，再用 MinHash 搭配 LSH 處理近似重複，最後用語意嵌入去重處理詞彙方法漏掉的改寫，因為每一層都能抓到較便宜那一層抓不到的東西。

### Q：面對資料汙染，你如何讓評估結果保持誠實？

**強答案：**
陷阱在於，單純的 n-gram 去汙染並不夠。一個著名的成果顯示，測試項目經過改寫或翻譯的版本能直接溜過 n-gram 比對，而在那些資料上訓練的模型可以把某個基準測試過度擬合到接近前沿的分數，甚至連強模型生成的合成資料回頭一驗也受到汙染。所以我會用語意方法來去汙染，也就是嵌入相似度搜尋加上一個 LLM 裁決者，而不只是字串重疊，並且在每次評估之前都對我所有的評估集執行一遍。更廣義來說，我會盡可能偏好保留集或新近發布的測試集，把任何基準測試都當成可能已被汙染來看待，並在真正重要的決策上倚賴我自己的黃金資料，因為那是我唯一能保證模型沒看過的那一組。
