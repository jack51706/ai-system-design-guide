# 分塊策略

分塊是把一份文件切分成多個離散片段以供檢索的過程。生產環境的管線已經不再使用盲目的固定大小切分，而是改用**結構感知與語意片段**，而像 late chunking 與情境前綴這類較新的技術，如今也已成為主流工具箱的一部分。

## 檢索與上下文之間的張力 {#tension}

| 面向 | 小區塊（100t） | 大區塊（1000t） |
|--------|---------------------|----------------------|
| **精確度** | 高（精準命中） | 低（被稀釋） |
| **上下文** | 差（句子被切斷） | 豐富（周邊資訊完整） |
| **儲存** | 高（向量較多） | 低（向量較少） |
| **延遲** | 低（搜尋快速） | 高（檢索負擔重） |

**法則**：較小的區塊較利於*尋找*，較大的區塊則較利於*思考*。請使用**階層式分塊**來同時兼得兩者。

## 遞迴結構切分 {#recursive}

我們不在每 500 個字元就硬切一刀，而是在邏輯邊界處進行切分：
`[Double Newline] > [Single Newline] > [Period] > [Space]`。

**最佳實務**：使用 **Markdown 感知切分**。如果文件中含有 `#` 標題，請確保把該標題前綴到*每一個*子區塊，以保留上下文（情境分塊，Contextual Chunking）。

## 語意分塊 {#semantic}

語意分塊會運用一個嵌入模型來偵測「主題轉移」。

1. 把文字切分成個別的句子。
2. 只要句子之間的嵌入相似度維持在某個門檻之上（例如 0.82），就把這些句子歸為同一群。
3. 一旦相似度下降，就另起一個新的區塊。

**細微之處**：生產環境的管線愈來愈常使用 **Cross-Encoder 分段器（Cross-Encoder Segmenters）**。一個極小的模型會掃描文字，並在每個語意斷點預測出一個「分隔符 token」。這比用餘弦相似度設門檻的做法精確 10 倍。

## 階層式（父子）分塊 {#hierarchical}

這是生產環境 RAG 的業界標準做法。

- **流程**：
  1. 建立大小為 1,500 tokens 的「父」區塊。
  2. 把每個父區塊再細分成 5 個大小為 300 tokens 的「子」區塊。
  3. **只對子區塊建立索引**。
  4. 在檢索時，如果某個子區塊命中，就**把完整的父區塊上下文回傳給 LLM**。
- **為什麼？**：子區塊夠小，向量資料庫容易命中。父區塊則提供足夠的上下文，讓 LLM 能夠真正正確地推理，而不會出現「上下文被切斷」所導致的幻覺。

## 內容專屬策略

### 1. 程式碼分塊
- **策略**：使用 AST（抽象語法樹，Abstract Syntax Tree）解析。
- **法則**：絕對不要在函式主體中間切開。讓 imports 與類別宣告與其方法保持在一起。

### 2. 表格分塊
- **策略**：使用 Markdown 格式來表示表格。
- **現代做法**：「摘要式表格（Summarized Tables）」。在向量資料庫中儲存該表格的自然語言摘要，但回傳給 LLM 的卻是完整的 Markdown 表格。

### 3. PDF／版面配置分塊
- **策略**：使用**視覺語言模型（Vision-Language Model, VLM）**進行前處理（例如 ColPali）。
- **細微之處**：不要只儲存文字，而是儲存能代表頁面*位置版面配置*的嵌入，以確保圖表與側欄不會被混進正文之中。

## 面試問題 {#interview-questions}

### Q：為什麼帶重疊的固定大小分塊對生產系統而言是有問題的？

**好的回答：**
固定大小分塊是「對內容無感」的。它經常在語意未完之處就把句子切斷、把數學方程式拆散，並把標題與其描述文字分隔開來。雖然「重疊」（例如 10%）可以藉由讓相鄰區塊之間重複 10% 的文字來緩解這個問題，但它並沒有解決核心問題：模型的注意力仍被迫要從片段化的字串中去重建語意。現代管線之所以偏好**語意或邏輯分塊**，是因為它能確保每個向量都代表一個「完整的語意單元」，進而帶來顯著更高的檢索精確度。

### Q：什麼是「情境式檢索（Contextual Retrieval）」（也就是 Anthropic 的做法）？

**好的回答：**
情境式檢索的做法是，在對每個區塊進行嵌入之前，先在區塊前面前綴一句全域情境。舉例來說，如果某個區塊講的是「電池續航力」，但它其實出自一份「2025 Model X 無人機」的手冊，那麼就會把 `[Drone_Model_X_Manual]:` 這段文字加到該區塊上。這能確保「電池續航力」的向量會受到「無人機」這個情境的影響，避免它在處理「手機電池」這類查詢時被誤檢索出來。
