# FinOps 與 Token 經濟學

本章談的是大規模運行 LLM 的**經濟學與紀律**：如何建模、歸因、編列預算，並從結構上降低 AI 支出。這不是重述推論內部機制；想了解戰術層級的調整手段（量化、批次處理、推測式解碼、KV cache），請參閱[成本優化實戰手冊](../04-inference-optimization/07-cost-optimization-playbook.md)以及[推論章節](../04-inference-optimization/01-inference-fundamentals.md)。

貫穿整章的核心發現，出自 Datadog 的 2026 State of AI Engineering：**系統提示約佔輸入 token 的 69%，但只有約 28% 的呼叫使用提示快取。** 在多數生產環境堆疊中，最大、最省力的成本槓桿正閒置未用。AI 產品同時是一門有銷貨成本的生意，而非零邊際成本的 SaaS，因此利潤思維如今已成為一項工程課題。

## 成本模型 {#the-cost-model}

定價以每百萬 token 報價，並區分輸入與輸出，而**輸出成本明顯高於輸入**，通常是 3 到 5 倍，有時更高，因為生成是自迴歸且受限於運算的，而輸入則是一次平行的 prefill 過程。（各模型的當前價格請見[定價與成本](../02-model-landscape/03-pricing-and-costs.md)；價格貶值很快，所以要對*結構*建模，而不是去算到分。）

每一次請求都可分解為層層堆疊的支出。把每一層分開建模，正是讓成本變得可預測、可歸因的關鍵：

| 層 | 由什麼驅動 | 行為 | 主要槓桿 |
|-------|-----------|----------|---------------|
| 系統提示 / 指令 | 固定鷹架、工具定義、few-shot | 約佔輸入 token 的 69%；若未快取則每次呼叫都付費 | 提示快取 |
| 檢索 / 上下文 token | RAG 區塊、注入的文件、長上下文 | 隨 k 與區塊大小擴展；可能遠超其他一切 | RAG 與長上下文之爭；區塊預算 |
| 對話 / 記憶 | 聊天歷史、代理草稿區 | 若不做摘要則無限增長 | 視窗化、摘要、壓縮 |
| 模型層級 | 前沿 vs 中階 vs 小型 / 自架 | 各層級之間落差達 10 到 100 倍 | 適當選型、級聯、路由 |
| 輸出長度 | 冗長度、格式、max_tokens | 以較高的輸出費率計費 | `max_tokens` 上限、精簡輸出契約 |
| 推理 / 思考 token | 延伸思考模式 | 以輸出費率計費，在回應中看不見 | 依任務複雜度控管思考 |
| 重試 / 額外開銷 | 暫時性錯誤、防護機制重跑 | 失敗時倍增 | 有界重試、斷路器 |
| 代理多步驟 | 規劃-行動-觀察迴圈、子代理 | 每一步都把整個堆疊倍增 | 步數上限、每次執行的預算 |

**推理模型與代理都是成本倍增因子，而這兩者最糟，因為它們是隱形的。** 延伸思考的 token 以輸出費率計費，卻不會出現在回應中，所以一次「簡短」的呼叫，可能比它可見的輸出所暗示的成本高出一個數量級；已發表的分析指出，倍率依任務從約 3 倍到約 15 倍不等。代理則在每一步都把*整個* token 堆疊倍增，因此正確的衡量單位是**每任務成本，而非每呼叫成本**。已報告的區間：一個聊天回合是幾美分，而一個代理式的多步驟任務則可能從幾十美分跑到好幾美元。

一條可供教學的每請求成本公式：

```
cost = Σ(layer_input_tokens × in_rate) + (output_tokens + reasoning_tokens) × out_rate
cost_per_task = cost_per_request × expected_steps × (1 + retry_rate)
```

並把快取當作對可快取輸入比例的折扣套用上去。

## 快取：頭號成本槓桿 {#caching-the-top-cost-lever}

這是頭號槓桿，正是因為那項核心統計：佔輸入 token 約 69% 的那一層（系統提示）是靜態的、最適合快取，但只有約 28% 的呼叫快取它。潛力與實際之間的落差，就是現成最大、最便宜的省錢空間。

**供應商前綴快取**讓你能對重複的提示前綴付出大幅折扣的費用。已報告的折扣範圍大致從約 50%（OpenAI，超過門檻後自動套用，無寫入費）到約 90%（Anthropic，需明確設定 `cache_control` 斷點，並有少量寫入溢價，讀取個幾次後便能打平），再到約 75%（Google）。教學時要明白點出的關鍵但書是：那個「節省 50% 到 95%」的標題，只適用於**被快取的前綴**，而非整張帳單。

如何實際拿到這份節省（紀律所在）：
- **提示順序由靜態排到動態。** 把系統指令、工具定義與 few-shot 範例放在最前面（穩定前綴），把使用者查詢放在最後。精確前綴比對意味著，靠近開頭的任何更動都會讓下游整個快取失效。
- **穩定前綴。** 快取區域內不要有時間戳記、請求 ID 或每次呼叫的 nonce；固定工具定義的順序。
- **留意 TTL 經濟學。** 在有寫入溢價的場合，快取只有在讀取次數越過打平門檻後才划算，所以爆量、低重用的流量可能得不到好處。
- **把每次呼叫的快取命中率當作一級指標來儀表化。** 改寫提示、升級模型版本、重新排序，都會悄悄拉低命中率。

有別於前綴快取，**精確比對與語意快取**會為重複的*查詢*提供整份回應。精確比對以字面請求為鍵（便宜、零誤命中）；語意快取則嵌入查詢，並對相似的命中提供已快取的回應（在自然語言流量上命中率更高，但有值得防範的誤命中風險）。把它們分層疊起來：先精確比對、再語意、再前綴，並逐層衡量命中率。從概念上講，這就是把[KV Cache 與上下文快取](../04-inference-optimization/02-kv-cache-and-context-caching.md)所述的同一套 KV 重用，在計費層上加以變現。

## 批次與非同步經濟學 {#batch-and-async-economics}

OpenAI 與 Anthropic 都提供**批次處理約 50% 的折扣**（輸入與輸出皆然），屬非同步，完成時限約為 24 小時。決策規則很簡單：**只要沒有人或系統在等這個 token，就用批次。** 高價值的批次工作負載包括評估與回歸測試套件、大量分類與標註、語料級的摘要與文件處理、提示或模型變更後的回填，以及對提示變體做 A/B 測試。對一個產品的整個離線層而言，不做批次等於把一半的錢留在桌上。

第三條路是**預配 / 預留吞吐量**（AWS Bedrock Provisioned Throughput、Azure OpenAI PTU）：以每小時費率預留容量，無論用量多寡都計費，據報在持續性工作負載上可省下約 15% 到 70%，但只有在高且可預測的使用率下才划算。其心智模型與雲端運算如出一轍：對尖峰或不確定的需求採用按 token 計費（含批次），一旦使用率高且穩定，就改用預留容量。

## FinOps 紀律 {#the-finops-discipline}

FinOps Foundation 的框架是：在許多部署中，**推論佔 GenAI 總支出的 80% 到 90%**，因此這套紀律聚焦於每請求的推論經濟學，而非訓練。其運營核心：

- **歸因。** 為每一次呼叫貼上團隊、功能、客戶 / 租戶、模型、路由與環境的標籤。技術上的促成者，是擺在 API 前面、能辨識每次呼叫來源的 token 代理或[閘道](03-ai-gateways-and-model-routing.md)。沒有歸因，就無法計算單位經濟，也看不出哪些使用情境賺回了自己的成本。
- **先 showback，再 chargeback。** 從可見性儀表板開始（按供應商、按模型、按團隊、按租戶，搭配每日預測與尖峰警示），等標籤可信之後，再進階到向各團隊計費。
- **單位經濟。** 追蹤每使用者、每對話、每解決工單或案件的成本，以及 AI 成本佔營收與佔毛利的百分比。把 AI 產品當成一門有銷貨成本的生意來教。
- **利潤現實。** 已報告的快照顯示，AI 產品的毛利大約比傳統 SaaS 的 80% 到 90% 低了約 25 到 30 個百分點，因為每一次請求都有變動的 token 成本。這正是為什麼**以成果計價**（按解決工單、按完成任務）正在興起，並出現像是「每解決一張客服工單收取固定價格」這類已報告的定錨。當務之急是：在你按解決量定價之前，先弄清楚你每次解決的成本。
- **工具。** 閘道提供即時的每請求控制與支出上限；FinOps 平台（Helicone、Vantage、Finout、Amnic，以及各種雲端成本工具）則提供跨雲的分攤與 chargeback。成熟的堆疊兩者並用。在以某個工具作為標準之前，先確認它目前的狀態；這個品類更迭很快。

## 結構性成本決策 {#structural-cost-decisions}

這些架構層級的選擇能讓成本變動 2 到 50 倍，遠超每呼叫的微調：

- **適當選型、級聯與路由。** 路由到能跨過品質門檻的最便宜模型，只在信心不足時才升級。據報在約 95% 的品質保留下可節省 45% 到 85%（FrugalGPT 是經典參考），而升級率就是那個即時變動的成本變數。請參閱 [AI 閘道與模型路由](03-ai-gateways-and-model-routing.md)。
- **自架 vs API 的損益平衡點。** 據報相對於前沿 API 的損益平衡點，落在每月幾千萬到上億 token 的高位，但承重的警告在於隱藏成本：純粹的 GPU 租金只佔真實成本的 30% 到 40%，所以要套用約 2.5 到 3 倍的乘數，而工程人力往往超過基礎設施。對 2026 年的多數團隊而言，一旦把整個堆疊算進去，受管 API 還是比較便宜；自架只有在量大、可預測、利用率高的情況下，或基於資料落地的理由時才勝出。請參閱 [LLM 基礎設施](01-llm-infrastructure.md)。
- **RAG vs 長上下文。** 檢索的每查詢成本遠比塞一個長上下文便宜，因為你只為幾個相關區塊付費，而不是一個巨大的提示。長上下文在小型靜態文件集上勝出；RAG 則在大型或頻繁變動的語料、以及高查詢量上勝出。請參閱 [RAG 基礎](../06-retrieval-systems/01-rag-fundamentals.md)。
- **蒸餾。** 把一個小模型微調到在鎖定的評估上與前沿模型相差幾個準確度百分點之內，據報可把每 token 成本砍掉 5 到 40 倍，在高量下幾週到幾個月就能回本；它在狹窄、高量的任務上勝出，在開放式的長尾工作上則失靈。請參閱[知識蒸餾](../03-training-and-adaptation/05-knowledge-distillation.md)與[蒸餾案例研究](../16-case-studies/19-customer-distillation-pipeline.md)。
- **輸出與提示工程。** 精簡的輸出契約、`max_tokens` 上限、結構化輸出，以及在模型可靠之後修剪 few-shot 範例，據報能在品質損失極小的情況下削減 20% 到 40% 的 token。

## 成本反面模式 {#cost-anti-patterns}

| 反面模式 | 機制 | 修正 |
|--------------|-----------|-----|
| 不做快取 | 每次呼叫都重新為靜態的系統提示（約佔輸入的 69%）付費 | 穩定前綴加上供應商前綴快取 |
| 模型過大 | 在小模型就能處理的任務上用前沿模型 | 適當選型、級聯、路由 |
| 輸出無上限 | 沒有 `max_tokens`、格式冗長 | 設上限與精簡輸出契約 |
| 預設開啟推理 | 對瑣碎任務也用延伸思考 | 依任務複雜度控管思考 |
| 重試風暴 | 暫時性錯誤觸發無界的重試 | 有界重試與斷路器 |
| 失控的代理迴圈 | 規劃-行動迴圈永不終止；工具錯誤被解讀為「重試」 | 在迴圈內設置硬性的步數、token 與重試上限 |
| 記憶無上限 | 歷史不斷累積卻不做摘要 | 視窗化與摘要 |
| 長上下文塞料 | 把巨大的上下文當作預設的檢索方式 | 對大型或變動的語料改用 RAG |
| 沒有歸因 | 未貼標籤的共享支出 | 閘道 token 代理加上標籤 |
| 用即時處理做離線工作 | 對評估、回填、標註使用同步 API | 批次 API |

已報告的真實事故讓代理迴圈那一列變得具體：失控的代理曾在無人察覺的情況下，於一個週末燒掉數萬美元。防線是迴圈內的硬性上限，而非事後的警示。

## 面試問題 {#interview-questions}

### Q：你的 LLM 帳單在流量持平的情況下，月對月翻了一倍。你會如何找出並修正？

**強答案：**
首先是歸因：如果不是透過閘道或代理把每次呼叫都按功能、團隊、模型與路由貼上標籤，那這就是第一個要修的地方，因為你無法除錯你看不見的東西。有了歸因，我會把支出拆解成各個 token 支出層，去找那些常見的元凶：一次破壞了快取命中率的提示變更（系統提示約佔輸入 token 的 69%，所以快取退化影響巨大）、對簡單任務也開了延伸思考（以輸出費率計費，且在回應中隱形）、步數悄悄攀升的代理迴圈、無上限的輸出或對話歷史，或是一場重試風暴。投報率最高的修正幾乎總是先恢復提示快取，接著是適當選型與限制輸出。我也會把任何離線工作（評估、回填）移到批次 API 以省下約一半，並設定每功能的預算搭配尖峰警示，好讓下一次翻倍在第一天就呼叫到某個人。

### Q：為什麼 AI 產品的毛利比 SaaS 差，而工程師對此能做什麼？

**強答案：**
因為每一次請求都帶有變動的 token 成本，所以 AI 產品的表現比較像一門有銷貨成本的生意，而非零邊際成本的軟體；據報毛利大約比典型 SaaS 低 25 到 30 個百分點。工程師從兩條戰線進攻。結構上：快取靜態的提示前綴、對模型適當選型與級聯、偏好 RAG 而非長上下文塞料、把高量的狹窄任務蒸餾到小模型上，並把離線層做批次。運營上：把單位經濟儀表化（每對話、每解決成果的成本），好讓定價能朝以成果計價的模式移動，而那只有在你弄清楚每次解決的成本時才行得通。這些成本槓桿是工程的責任，不只是財務的。
