# 2026 年的工具使用與電腦代理全景

AI 代理與外部世界互動的方式已經歷劇烈的轉變。在 2024 年，「工具使用」指的是模型發出一個 JSON 函式呼叫，再由你的後端執行。如今我們有了功能完整的自主代理，能夠複製程式碼倉庫、執行 shell 命令、透過螢幕截圖控制桌面，還能在 WhatsApp 上傳訊息給你，這一切都透過像 MCP 這樣的標準化協定來編排。本章將描繪這些工具的全景、它們的架構，以及讓彼此有所區別的設計決策。

2026 年的工具使用生態系已收斂為四種架構模式，每一種都針對不同程度的自主性、安全性與整合深度進行了最佳化：

## 類別分類

### 1. 本地端代理（自架、由使用者掌控）

在使用者自己硬體上執行的代理。LLM 呼叫可能送往雲端，但代理程序、記憶與工具執行都在本地端。

**關鍵特性：**
- 在使用者機器上擁有完整檔案系統存取權
- 持久化記憶儲存在本地（SQLite、JSON、Markdown）
- 使用者擁有所有資料，沒有廠商鎖定
- 安全責任完全落在操作者身上

**範例：** OpenClaw、Open Interpreter、本地端 OpenHands 部署

### 2. 雲端代理（廠商代管、由 API 驅動）

在廠商管理的雲端環境中執行的代理。程式碼執行發生在沙箱化的 VM 或容器中。

**關鍵特性：**
- 沙箱化執行（Docker、Firecracker VM、E2B）
- 沒有本地端檔案系統存取（在複製的程式碼倉庫上運作）
- 廠商負責擴展、安全與基礎設施
- 按使用量計費或訂閱制定價

**範例：** Claude Code（雲端模式）、OpenAI Codex、Google Jules、OpenHands Cloud

### 3. IDE 代理（整合於編輯器、具情境感知）

直接嵌入程式碼編輯器中的代理。它們對專案結構、開啟的檔案以及編輯器狀態有深入的理解。

**關鍵特性：**
- 與編輯器 UI 緊密整合（行內差異對比、Tab 自動補全）
- 透過嵌入或 AST 解析建立程式碼庫索引
- 在分支上非同步運作的背景代理
- 為開發者工作流程最佳化，而非通用自動化

**範例：** Cursor（Agent Mode + Background Agents）、Windsurf（Cascade）、Cline、GitHub Copilot、Google Antigravity（以 Gemini 3 為核心、代理優先的工作空間，帶有多代理管理者檢視畫面，為 Gemini CLI 的後繼者）

### 4. 電腦使用代理（以視覺為基礎、由 GUI 驅動）

以人類的方式與軟體互動的代理，也就是透過觀看螢幕截圖並進行點擊。

**關鍵特性：**
- 模型看著螢幕截圖，決定滑鼠/鍵盤動作
- 可與任何應用程式搭配運作（不需要 API）
- 延遲較高（截圖到動作的迴圈每一步需 1 至 3 秒）
- 為了安全需要沙箱化環境（VM + VNC）

**範例：** Claude Computer Use API、Open Interpreter Computer API

## OpenClaw：爆紅的個人 AI 代理

### 它是什麼

OpenClaw 是一套自架、開源的個人 AI 助理，由奧地利開發者 Peter Steinberger 所打造。最初於 2025 年 11 月以「Clawdbot」之名發布，並於 2026 年 1 月更名為 OpenClaw。它在不到五個月的時間內，從 0 暴增到 346,000 個 GitHub stars，並於 2026 年 3 月 3 日超越 React，成為 GitHub 上 star 數最多的軟體專案。

**數據一覽（2026 年 5 月）：**
- 346,000+ GitHub stars
- 320 萬名活躍使用者
- 500,000+ 個運行中的實例
- ClawHub 上有 44,000+ 個社群技能
- 專案網站每月 3,800 萬名訪客
- 24+ 個訊息平台整合

### 它如何運作

OpenClaw 的架構有六個核心元件：

```
+-------------------------------------------------------------------+
|                      OpenClaw Architecture                        |
+-------------------------------------------------------------------+
|                                                                   |
|  +-----------+     +-----------+     +----------+                 |
|  |  Gateway  |---->|  LLM      |---->| PI Agent |                 |
|  |           |     |  (Brain)  |     | (Exec)   |                 |
|  +-----------+     +-----------+     +----------+                 |
|       ^                  |                |                       |
|       |                  v                v                       |
|  +-----------+     +-----------+     +----------+                 |
|  | Channels  |     | SOUL.md   |     | Skills   |                 |
|  | (24+)     |     | (Identity)|     | (44K+)   |                 |
|  +-----------+     +-----------+     +----------+                 |
|                          |                                        |
|                          v                                        |
|                    +-----------+                                  |
|                    | Memories  |                                  |
|                    | (Persist) |                                  |
|                    +-----------+                                  |
+-------------------------------------------------------------------+
```

**1. Gateway（閘道）**：訊息進出層。連接到 WhatsApp（透過 Baileys）、Telegram、Discord、Slack、Signal、iMessage、Microsoft Teams、Matrix，以及 16+ 個其他平台。支援私訊與群組對話，並以提及（mention）為基礎觸發啟用。

**2. LLM（大腦）**：在設計上與模型無關。支援 GPT-4o、Claude、Gemini、DeepSeek，或透過 Ollama 使用本地模型。由使用者挑選模型，架構本身並不在意。

**3. PI Agent（Process Interactor，程序互動器）**：一個小型執行環境，讓 LLM 能在主機系統上建立、編輯、執行與刪除檔案。LLM 產生程式碼，PI Agent 將其儲存後執行。這是代理的「雙手」。

**4. SOUL.md（身分層）**：一個純 Markdown 檔案，定義代理的個性、溝通風格、價值觀與行為防護機制。在工作階段開始時載入並注入到系統提示中。每個代理實例都會先讀取 SOUL.md，它「把自己讀進存在」。

**5. Skills（外掛系統）**：賦予代理新能力的擴充功能。ClawHub 上存在超過 44,000 個社群技能。技能遵循 AgentSkills 規範，可以是套裝、工作空間本地，或全域安裝。

**6. Memories（持久化情境）**：儲存在本地端的長期記憶。代理會跨對話累積關於使用者的情境。結合 SOUL.md，這讓每個代理在所有訊息平台上都擁有一致的個性。

### 工作空間檔案

| 檔案 | 用途 |
|------|---------|
| `SOUL.md` | 代理的個性、語氣、價值觀、防護機制 |
| `AGENTS.md` | 操作指示、工具設定 |
| `HEARTBEAT.md` | 排程的自主動作（類似 cron） |
| `Memories/` | 跨對話的持久化情境 |

### 安全疑慮

OpenClaw 的快速成長已超出其安全實務所能負荷。截至 2026 年 5 月，超過 135,000 個實例暴露在公開網際網路上，其中許多仍使用預設設定。ClawHub 技能市集的安全監督極少：技能是搭配選用 TypeScript 的 Markdown，易於建立與安裝，也易於濫用。對於任何要在生產環境中部署 OpenClaw 的人來說，這是一個關鍵的設計考量。

## OpenHands：自主開發者代理

### 它是什麼

OpenHands（前身為 OpenDevin）是一套開源的自主 AI 軟體工程師。採用 MIT 授權，能夠修改程式碼、執行命令、瀏覽網頁，並與 API 互動。與只會建議程式碼片段的工具不同，OpenHands 會複製程式碼倉庫、執行終端機命令、運行測試，並在沙箱化的 Docker 容器內除錯。

### 架構：事件串流 + 沙箱化執行環境

```
+-------------------------------------------------------------------+
|                     OpenHands Architecture                        |
+-------------------------------------------------------------------+
|                                                                   |
|  +------------------+                                             |
|  |   User / API     |                                             |
|  +--------+---------+                                             |
|           |                                                       |
|           v                                                       |
|  +--------+---------+     +------------------+                    |
|  |  Agent Controller |<--->|  Event Stream    |                   |
|  |  (CodeAct 1.0)   |     |  Hub             |                   |
|  +--------+---------+     +--------+---------+                    |
|           |                        |                              |
|           v                        v                              |
|  +--------+---------+     +--------+---------+                    |
|  |  Action Dispatch  |     |  Observation     |                   |
|  |                   |     |  Collector       |                   |
|  |  - CmdRunAction   |     |                  |                   |
|  |  - FileWriteAction|     |  - CmdOutput     |                   |
|  |  - BrowseURLAction|     |  - FileContent   |                   |
|  |  - CodeAction     |     |  - BrowserState  |                   |
|  +--------+---------+     +------------------+                    |
|           |                                                       |
|           v                                                       |
|  +--------+--------------------------------------------------+    |
|  |              Docker Sandbox (Per Session)                 |    |
|  |                                                           |    |
|  |  +----------+  +----------+  +----------+                |    |
|  |  | Terminal  |  |  Python  |  | Browser  |                |    |
|  |  | (bash)   |  | (stateful)|  | (BrowserGym)             |    |
|  |  +----------+  +----------+  +----------+                |    |
|  +-----------------------------------------------------------+    |
+-------------------------------------------------------------------+
```

**關鍵架構決策：**
- **事件串流架構**：所有代理與環境之間的互動都以型別化事件的形式流經一個中央樞紐。代理分析對話狀態並產生動作（Actions），沙箱則產生觀察結果（Observations）。
- **每個工作階段獨立的 Docker 容器**：每個工作階段都會取得自己的隔離容器，擁有完整的作業系統能力。容器與主機相互隔絕。
- **CodeAct 1.0**：預設的代理範本。將 LLM 推理嵌入一個統一的程式設計控制平面，並維持工作階段層級的專案情境。
- **BrowserGym 整合**：代理可透過宣告式原語（DOM 操作、導覽）進行瀏覽器自動化。
- **SDK 可組合性**：OpenHands SDK 是一個 Python 函式庫。你可以用程式碼定義代理、在本地端執行，或在雲端擴展到數千個。

**近期更新（v1.6.0，2026 年 3 月）：**
- 支援 Kubernetes 以編排代理工作階段
- Planning Mode beta，用於多步驟任務分解
- 來自 188+ 名貢獻者的 2,100+ 次貢獻

## Open Interpreter：本地端程式碼執行

### 它是什麼

Open Interpreter 是一套本地端程式碼執行代理，提供類似 ChatGPT 的終端機介面。Open Interpreter 不會顯示程式碼後請你執行，而是徵求許可後直接在你的機器上執行，並擁有對本地檔案的完整存取權。

### 架構

```
+-------------------------------------------------------------------+
|                  Open Interpreter Architecture                    |
+-------------------------------------------------------------------+
|                                                                   |
|  +------------------+                                             |
|  |  Terminal UI      |                                            |
|  |  (ChatGPT-like)  |                                            |
|  +--------+---------+                                             |
|           |                                                       |
|           v                                                       |
|  +--------+---------+     +------------------+                    |
|  |  Core Engine      |<--->|  LLM Provider   |                   |
|  |                   |     |  (100+ models)  |                    |
|  |  - NL to Code     |     |  GPT, Claude,   |                   |
|  |  - Permission     |     |  Ollama, LM     |                   |
|  |    Gate            |     |  Studio, etc.   |                   |
|  +--------+---------+     +------------------+                    |
|           |                                                       |
|           v                                                       |
|  +--------+---------+                                             |
|  |  Code Executor    |                                            |
|  |                   |                                            |
|  |  - Python         |                                            |
|  |  - JavaScript     |                                            |
|  |  - Shell/Bash     |                                            |
|  |  - AppleScript    |                                            |
|  +--------+---------+                                             |
|           |                                                       |
|           v                                                       |
|  +--------+---------+                                             |
|  |  Computer API     |                                            |
|  |  (GUI Control)    |                                            |
|  |                   |                                            |
|  |  - Screen capture |                                            |
|  |  - Mouse/Keyboard |                                            |
|  |  - Icon detection |                                            |
|  +-------------------+                                            |
+-------------------------------------------------------------------+
```

**關鍵特性：**
- **模型彈性**：可與 100+ 個 LLM 搭配運作。使用 GPT-4o 或 Claude 以獲得最大能力，或為了隱私而完全離線運行，搭配 Ollama 與 LM Studio。
- **許可閘門**：每次程式碼執行都需要使用者核准（在受信任的工作流程中可停用）。
- **Computer API**：除了程式碼執行之外，Open Interpreter 還能看到你的螢幕、辨識 UI 元素，並控制你的滑鼠與鍵盤，將它從一個程式碼直譯器提升為一個電腦自動化代理。
- **預設不沙箱化**：直接在主機上運行。這是為了追求最大能力而刻意做出的設計選擇，但這意味著一個糟糕的 LLM 輸出可能會損壞你的系統。Docker 沙箱化是選用的。

### 何時該使用 Open Interpreter

最適合用於資料分析、檔案操作與系統管理任務，這些情境下你想要一個對本地機器的對話式介面。不適合用於生產環境部署或不受信任的環境。

## Claude Computer Use：以視覺為基礎的自動化

### 它是什麼

Claude Computer Use 是 Anthropic 的一項 API 功能，讓 Claude 能透過螢幕截圖、滑鼠移動、鍵盤輸入與應用程式互動來控制桌面。它於 2024 年 10 月以 beta 形式推出，此後已大幅演進。截至 2026 年 5 月，Sonnet 4.6 在 OSWorld-Verified 上達到 72.5%，相較於推出時的 14.9% 大幅提升，而 Opus 4.7 在代理式程式設計基準測試上更進一步（SWE-bench Pro 達 64.3%）。

### 視覺到動作的迴圈

```
+-------------------------------------------------------------------+
|              Claude Computer Use: Vision-Action Loop               |
+-------------------------------------------------------------------+
|                                                                   |
|  Step 1: OBSERVE          Step 2: REASON          Step 3: ACT    |
|  +----------------+       +----------------+      +------------+ |
|  |  Take          |       |  Analyze       |      |  Execute   | |
|  |  Screenshot    |------>|  Screenshot    |----->|  Action    | |
|  |  (base64 PNG)  |       |  + Task Goal   |      |  (click,   | |
|  |                |       |  + History      |      |  type,     | |
|  +----------------+       +----------------+      |  scroll)   | |
|                                                    +------+-----+ |
|                                                           |       |
|          +------------------------------------------------+       |
|          |                                                        |
|          v                                                        |
|  +-------+--------+                                               |
|  |  Wait + Take   |                                               |
|  |  New Screenshot|-------> (Loop back to Step 1)                 |
|  +----------------+                                               |
|                                                                   |
+-------------------------------------------------------------------+
```

### 可用的工具

| 工具 | 能力 | 備註 |
|------|------------|-------|
| `computer` | 滑鼠、鍵盤、螢幕截圖 | 完整桌面 GUI 控制 |
| `bash` | 執行 shell 命令 | 跨回合的持久化工作階段 |
| `text_editor` | 讀取/寫入/編輯檔案 | 支援 view、create、str_replace |

### 2026 年的強化

- **Zoom Action（縮放動作）**：在點擊前以高解析度檢視小型 UI 元素。降低在密集介面上的誤點率。
- **可在 Claude Cowork 與 Claude Code 中使用**：提供給 Pro 與 Max 使用者的研究預覽，在執行破壞性動作前需要人類確認。
- **沙箱化最佳實務**：務必在沙箱化的 VM 中執行（Docker + VNC，或 E2B 雲端）。絕不要將電腦使用權限交給一台未沙箱化的主機。

| 日期 | OSWorld 分數 | 關鍵里程碑 |
|------|---------------|---------------|
| 2024 年 10 月 | 14.9% | Beta 推出（Claude 3.5 Sonnet） |
| 2025 年中 | ~40% | Claude 3.7 改進 |
| 2026 年第一季 | 72.5% | Sonnet 4.6、Zoom Action |

## Claude Code：終端機代理

### 它是什麼

Claude Code 是 Anthropic 的代理式程式設計工具，存在於終端機中。它會讀取你的程式碼庫、編輯檔案、執行命令，並與開發工具整合。它於 2025 年 5 月公開推出，並在 2026 年 2 月跨越 25 億美元 ARR。

### 架構

Claude Code 是一個 TypeScript 終端機代理，會循環經過三個階段：

```
+-------------------------------------------------------------------+
|                   Claude Code Agent Loop                          |
+-------------------------------------------------------------------+
|                                                                   |
|  +------------------+                                             |
|  |  1. GATHER       |  Read files, grep codebase, glob search,   |
|  |     CONTEXT      |  check git status, analyze structure        |
|  +--------+---------+                                             |
|           |                                                       |
|           v                                                       |
|  +--------+---------+                                             |
|  |  2. TAKE         |  Edit files, run bash, write new files,     |
|  |     ACTION       |  create commits, spawn subagents            |
|  +--------+---------+                                             |
|           |                                                       |
|           v                                                       |
|  +--------+---------+                                             |
|  |  3. VERIFY       |  Run tests, check build, review diffs,     |
|  |     RESULTS      |  validate output                            |
|  +--------+---------+                                             |
|           |                                                       |
|           +--------> (Loop back to Step 1 if not done)            |
|                                                                   |
+-------------------------------------------------------------------+

Built-in Tools: bash, read, write, edit, glob, grep, browser,
                subagent, notebook, web_search, web_fetch
```

**關鍵架構特性：**
- 一個代理迴圈，搭配豐富的工具調色盤
- 透過斜線命令與 CLAUDE.md 進行隨選技能載入
- 為長工作階段進行的上下文壓縮（1M+ token 的上下文視窗）
- 為平行工作流派生子代理
- 為平行分支執行進行的 worktree 隔離
- 權限治理（工具的允許/拒絕規則）
- 帶有相依圖的任務系統
- 用於自訂自動化的 Hooks（提交前/後、檔案變更）

## IDE 代理：Cursor、Windsurf、Cline

### Cursor

Cursor 是一個 VS Code 分支，具備深度 AI 整合。2.0 版（2026 年初）引入了：
- **Agent Mode**：使用 20 倍規模的強化學習進行多檔案編輯
- **Background Agents**：在雲端 VM 中複製你的程式碼倉庫，自主運作，完成後開啟 PR
- **Mission Control**：用於管理平行代理工作流程的儀表板
- **市場表現**：年化營收 20 億美元、200 萬名以上使用者、100 萬名以上付費客戶，獲半數財星 500 大企業採用

### Windsurf

Windsurf（原名 Codeium，於 2025 年 7 月由 Cognition 以 2.5 億美元收購）具備：
- **Cascade**：多步驟 AI 代理，會分析專案結構、協調跨檔案的變更，並從錯誤中自我恢復
- **專有模型**：SWE-1.5（比 Sonnet 4.5 快 13 倍）與 Fast Context
- **Codemaps**：由 AI 驅動的視覺化程式碼導覽
- **跨 IDE 外掛**：可用於 40+ 個 IDE（JetBrains、Vim、NeoVim、XCode）

### Cline

Cline 是一個 VS Code 擴充功能，以完整代理而非自動補全工具的形式運作。它會採取一連串步驟、評估結果、修正自己的錯誤，然後繼續。比 Cursor 或 Windsurf 更為自主，但精緻度較低。

## 市場趨勢與採用情況（2026）

### 數據

- **MCP 生態系**：10,000+ 個活躍伺服器，每月 9,700 萬次 SDK 下載
- **Gartner 預測**：到 2026 年底，40% 的企業應用程式將納入 AI 代理（相較於 2025 年初不到 5%）
- **OpenClaw**：史上最快達到 30 萬 GitHub stars 的專案（不到 5 個月）
- **Claude Code**：到 2026 年 2 月達到 25 億美元 ARR，史上最快達到 10 億美元的企業軟體產品
- **Cursor**：年化營收 20 億美元，半數財星 500 大企業

### 關鍵趨勢

**1. 代理類型的收斂**：本地端、雲端與 IDE 代理之間的界線正在消融。Claude Code 在本地端執行，但使用雲端模型。Cursor 的 Background Agents 在雲端執行。OpenClaw 可連接任何 LLM。趨勢正朝向一個能在任何環境中運作的通用代理架構發展。

**2. MCP 作為通用工具層**：MCP 已成為工具整合的標準，獲得 Anthropic、OpenAI、Google 以及數百家工具供應商採用。2026 年的路線圖聚焦於企業就緒度：身分傳播、工具預算、結構化錯誤語意，以及稽核軌跡。

**3. 沙箱化成為不可妥協的要求**：OpenClaw 的安全危機（135,000 個暴露的實例）已推動整個產業朝向預設沙箱化的架構發展。新代理被期望要開箱即提供隔離能力。

**4. 成本最佳化成為一等公民的考量**：Plan-and-Execute 模式（由一個有能力的模型進行規劃，較便宜的模型負責執行）可降低 90% 的成本。這是代理式版本的雲端成本最佳化。

**5. 背景與非同步代理**：Cursor 的 Background Agents 與 Claude Code 的子代理派生，代表了從同步、互動式代理轉向自主、非同步工作者的轉變，這些工作者會在完成後通知你。

## 系統設計面試切角

在系統設計面試中被問及工具使用代理時，聚焦於以下這些面向：

**1. 安全模型**：執行是否沙箱化？憑證如何管理？若 LLM 產生惡意程式碼會發生什麼事？（OpenClaw 的 AGPL 授權與未沙箱化執行，相較於 OpenHands 的 Docker 隔離，是絕佳的比較切入點。）

**2. 狀態管理**：代理如何跨工具呼叫維持情境？以工作階段為基礎（OpenHands）、持久化記憶（OpenClaw），還是以檔案為基礎（Claude Code 的 CLAUDE.md）？

**3. 工具發現**：靜態清單（舊做法）、透過 MCP 進行的動態發現，還是技能市集（OpenClaw ClawHub）？

**4. 延遲預算**：函式呼叫（每次工具呼叫 50 至 200 毫秒），相較於以視覺為基礎的自動化（每個截圖到動作的迴圈 1 至 3 秒）。這如何影響 UX？

**5. 失敗處理**：當一次工具呼叫失敗時會發生什麼事？重試？退而求其次？人類介入迴圈（Human-in-the-Loop）？在放棄之前要重試幾次？

## 面試題

### Q：你的團隊想打造一個內部 AI 助理。你應該基於 OpenClaw、OpenHands 來建構，還是用 Claude Code + MCP 客製化開發？

**有力的回答：**
這取決於使用情境與安全需求。OpenClaw 為帶有訊息整合的個人助理做了最佳化，如果目標是一個具有持久化個性的 Slack/Teams 機器人，那它就很理想。但它未沙箱化的執行與 AGPL 授權會帶來企業層面的疑慮。OpenHands 比較適合自主開發任務，它的 Docker 沙箱化與 MIT 授權對企業友善。對於一個客製化的內部工具而言，Claude Code 搭配 MCP 伺服器能給你最大的掌控權：你可以精確定義哪些工具可用、在自己的基礎設施中執行它們，並受益於 MCP 標準化的發現與授權機制。這個決策樹是：以訊息為優先？選 OpenClaw。開發自動化？選 OpenHands。客製化的企業工具？選 MCP + 你自己的代理迴圈。

### Q：你會如何設計一個讓非技術使用者能用 AI 自動化桌面任務的系統？

**有力的回答：**
我會使用以視覺為基礎的電腦使用模式（Claude Computer Use 或類似工具）。關鍵的設計決策有：(1) 務必在沙箱化的 VM 中執行，這樣代理就無法損壞使用者真正的機器。(2) 在任何破壞性動作之前實作一個人類介入迴圈（Human-in-the-Loop）的確認步驟，例如刪除檔案、提交表單、進行採購。(3) 使用 Zoom Action 模式來降低在密集 UI 上的誤點。(4) 設定 token/成本上限以防止失控的迴圈。(5) 將所有動作記錄為稽核軌跡。主要的取捨在於延遲，每個截圖到動作的步驟需要 1 至 3 秒，但這種做法不需要 API 就能與任何應用程式搭配運作。對於速度要求較高的工作流程，可將電腦使用與函式呼叫結合，用於具備 API 的應用程式。

### Q：為什麼 OpenClaw 的成長速度比史上任何開源專案都快？這告訴你關於市場的什麼資訊？

**有力的回答：**
三個因素。(1) **零摩擦上手**：OpenClaw 連接到人們已在使用的訊息平台（WhatsApp、Telegram）。使用者不需要學習一個新介面。(2) **SOUL.md 個人化**：能夠賦予你的代理一個客製化個性，創造了情感連結與病毒式傳播，人們會分享自己的代理。(3) **與模型無關的架構**：使用者不會被鎖定在單一 LLM 供應商，降低成本並增加彈性。市場訊號是代理的「介面」比底層模型更重要。人們想要的是能在他們所在之處與他們相遇的代理（訊息應用程式，而非 Web UI）。另一面是：沒有安全投資的快速成長會導致像 135,000 個暴露實例這樣的危機，這對任何開源代理專案都是一則警世故事。

### Q：比較 AI 代理的沙箱化與未沙箱化執行。你會在什麼時候選擇各自的方案？

**有力的回答：**
沙箱化（Docker/VM）：用於不受信任的程式碼執行、多租戶系統，或任何生產環境部署。OpenHands 在這方面做得很好，每個工作階段都會取得自己的 Docker 容器。取捨在於設定的複雜度與效能開銷。未沙箱化（主機存取）：僅用於單一使用者、受信任的環境，且使用者正在旁觀。Open Interpreter 與 OpenClaw 採取這種做法以追求最大能力。風險在於一個糟糕的 LLM 輸出可能損壞主機系統。2026 年的共識是預設沙箱化，並為進階使用者保留逃生出口。在面試中，務必提到沙箱邊界是一個安全決策，而不只是一個便利性決策。
