# 擴散語言模型

幾乎你用過的每一個 LLM 都是自迴歸（autoregressive，AR）的：它每次前向傳播只產生一個 token，由左到右逐一輸出。**擴散語言模型**走的是另一條路，這條路借鑑自影像生成：從一個完全被遮罩的序列出發，在少數幾個步驟內，對*所有位置平行地*反覆去噪。它換來的好處是吞吐量，已回報的數字超過每秒 1,000 個 token，比經過速度優化的 AR 模型高出一個數量級。代價是部分品質，而且成熟度還很早期。本章說明它們如何運作、在哪裡勝出，以及工程師在 2026 年實際上該拿它們來做什麼。

## 如何運作 {#how-they-work}

AR 模型把一個序列由左到右做因式分解：每個 token 只根據過去來預測，每次前向傳播產生一個。**遮罩擴散模型**則學會從一個被破壞（遮罩）的版本重建出整個序列，在少數幾個去噪步驟內一次精修所有位置，並採用**雙向**注意力（每個位置同時看得到過去與未來）。

最具代表性的參考文獻是 **LLaDA**（arXiv:2502.09992）。其機制，以可教學的層次來說：
- **正向過程：** 從乾淨文字出發；在噪音水準 `t` 下，以機率 `t` 把每個 token 替換成一個 `[MASK]`。在 `t=1` 時整個序列被完全遮罩。
- **反向過程：** 訓練一個 Transformer（一個「遮罩預測器」）來同時預測每個被遮罩位置上的原始 token，所優化的是一個概似度*下界*（而非 AR 模型所用的精確概似度）。
- **取樣：** 從完全遮罩開始，由 `t=1` 去噪到 `0`。在每一步，模型預測所有被遮罩的 token，並由一個**解遮罩排程（unmasking schedule）**決定要提交多少個信心最高的預測，以及要把多少個重新遮罩留給下一步。步數越少代表越快、品質越低；每步只解一個 token 會逼近 AR 品質，但會放棄速度。

平行解碼與雙向上下文直接由此自然產生，並且給了擴散一個值得注意的把戲：LLaDA 回報在反向任務（把一首詩倒著補完）上勝過 GPT-4o，因為其目標並未被鎖定在由左到右的順序上。

最乾淨的心智模型不是「AR 對擴散」，而是兩者之間的一種**插值**，這由 Block Diffusion（BD3-LMs，arXiv:2503.09573）所確立：把序列切成多個區塊，跨區塊走 AR、區塊內走擴散。這找回了純擴散難以兼顧的兩件事，KV cache 與可變長度輸出，而區塊大小就是那個旋鈕（區塊大小為 1 即是 AR，區塊大小等於整個序列即是純擴散）。幾乎每一個 2026 年的快速系統都會推出某種區塊或半 AR 變體。

## 速度優勢與取捨 {#the-speed-advantage-and-the-tradeoff}

速度提升只來自一個地方：**每次前向傳播所提交的 token 數量。** AR 產生一個；擴散在少數幾步內提交許多個，因此當模型能夠有信心地一次解遮罩數個位置時，實際時鐘吞吐量便上升。這是一個延遲與吞吐量的勝利，本質上並非品質上的勝利。

已回報的吞吐量（全都是廠商或基準測試的說法，條件各異）：商用擴散模型回報在單一資料中心 GPU 上達到約每秒 700 到 1,500 個 token 的量級，其製造商把它框定為比經過速度優化的 AR 基線快上數倍（在最佳情況的程式碼補全路徑上可達約 10 倍，在推理工作負載上則較接近約 5 倍）。對某一個商用模型的獨立量測把它放在每秒約 1,100 個 token 左右。

這個取捨是解遮罩排程所固有的：每步提交越多 token 越快，但可能產生局部不連貫的輸出，因此實務上最佳的品質來自每步解遮罩較少的 token，而這會拖慢速度。2026 年品質差距的現況，依回報所示：
- **在程式碼上，差距很小。** 擴散模型在程式碼補全基準測試上大致與同級別的 AR 模型旗鼓相當；程式碼是公認的最佳切入點。
- **在知識與困難推理上，差距較大**，而且沒有任何擴散模型屬於前沿級別。一個公認的理論逆風是，遮罩擴散優化的是概似度*下界*，而非精確概似度。

## 2026 年的全景 {#the-2026-landscape}

把版本、速度與價格都當成一份易腐的快照來看待。

| 系統 | 類型 | 備註（依回報） |
|--------|------|------------------|
| **Mercury / Mercury 2**（Inception Labs） | 封閉、先聚焦程式碼、後轉通用 | 最清楚的生產環境選項：OpenAI 相容 API 與一個雲端市集上架項目，每秒約 1,000 到 1,200 個 token，價格公開。被行銷為一個快速推理模型，這個說法有些分析師持異議，因為擴散在多步推理上有其限制；請在你自己的評估上驗證。 |
| **Gemini Diffusion**（Google） | 封閉展示 | 展示速度令人印象深刻；截至 2026 年中**沒有公開 API**。不要圍繞它做架構設計。 |
| **DiffusionGemma**（Google） | 開放權重 | 一個開放權重的 Gemma 家族擴散模型，回報在單一資料中心 GPU 上超過每秒 1,000 個 token。不要把它與 Gemini Diffusion 混為一談。 |
| **LLaDA / LLaDA 2.0** | 開放權重 | 那個奠基性的開放模型，以及一個後來的 MoE 後繼者；與同尺寸的 AR 模型有競爭力，但非前沿。 |
| **Dream 7B** | 開放權重 | 回報可與同尺寸的 AR 模型相當或超越，具備一個可調的步數旋鈕與強大的規劃能力（數獨、Countdown）；存在一個完全開放的程式碼變體。 |

## 混合做法：用擴散打草稿，用 AR 驗證 {#hybrids-draft-with-diffusion-verify-with-ar}

與生產環境最相關的研究把這兩個範式融合在一起：

- **擴散作為推測式草稿產生器。** 一個擴散模型平行提出一個 token 區塊，再由一個 AR 模型驗證它們，從而保留 AR 的目標分布，類比於[推測式解碼](03-speculative-decoding.md)。TiDAR（arXiv:2511.08923）回報每秒 token 數約為 AR 的 5 倍，同時收斂了品質差距，被定位為在延遲關鍵的用途上勝過標準推測式解碼。
- **區塊／半 AR 改造。** Fast-dLLM v2（arXiv:2509.26328）把一個既有的 AR 模型（Qwen）改造成一個區塊擴散模型，搭配逐區塊的 KV cache 與信心感知的平行解碼，這是從你手上已有的檢查點榨出可快取、快速擴散解碼的實用配方。
- **用於擴散推理的 RL。** d1（arXiv:2504.12216）是第一個用於遮罩擴散 LLM 的 GRPO 風格 RL（diffu-GRPO），把遮罩 SFT 與一個高效的 RL 階段配對起來，並回報在數學與規劃基準測試上相對於基礎擴散模型有一致的提升。

近期可能的生產環境路徑並非純擴散，而是這些混合做法，它們找回了 KV cache 與品質，同時保住大部分的速度。

## 擴散在哪裡勝出，AR 又在哪裡仍然勝出 {#where-diffusion-wins-and-where-ar-still-wins}

**擴散的優勢（結構性的，不只是炒作）：**
- **低延遲、高吞吐量的服務**，這是頭號用例，在往返延遲主導一切的場景（互動式程式撰寫、自動補全、即時代理）。
- **程式碼**，這裡的生成是反覆來回的精修，而非嚴格循序的。
- **填空與多區域編輯**，這是遮罩目標的原生能力：遮罩相關位置並一次重新生成它們全部（舉例來說，在一個函式簽章、它的 docstring、呼叫處與測試之間一致地新增一個參數，一次完成）而不是去編排多個 AR 呼叫。
- **反向以及對順序不敏感的推理與規劃**，這裡雙向上下文有幫助。

**2026 年 AR 仍然勝出之處：** 長上下文與代理式／工具使用的工作流程（一份 2026 年的實況查核研究記錄了擴散在長上下文處理、KV cache 效率與函式呼叫上的弱點）；在知識與困難推理上的前沿品質；以及極長或可變長度的生成，這是純擴散處理得很差、而區塊／半 AR 變體正是為了修補它而存在的。AR 還擁有多年的服務堆疊與 KV cache 優化，這是擴散的生態系尚未追上的。

## 成熟度與今天該怎麼做 {#maturity-and-what-to-do-today}

老實的判讀：大致仍處早期，在一個快速演進的開放研究基礎上，覆著薄薄一層真正的商用產品。工程師該做什麼：

1. **不要替換掉你的 AR 前沿模型**，對於品質關鍵、長上下文或代理式的工作而言。
2. **要在延遲與吞吐量主導、且輸出屬於短到中等且結構化的場景試行擴散**，程式碼補全與填空、批量轉換、結構化擷取。一個商用的 OpenAI 相容擴散 API 是阻力最低的 A/B 測試；若你需要自行託管，則用一個開放權重模型。
3. **盯著那些混合做法**，而不只是純擴散，因為「區塊／半 AR」加上「擴散打草稿、AR 驗證」才是可能的生產環境路徑。
4. **在你自己的任務上，用實際的解碼設定做基準測試。** 你得到的品質是「你解遮罩得多激進」的直接函數，而廠商的每秒 token 數假設了激進的設定。
5. **對「推理擴散模型」的行銷抱持懷疑**，並在你自己的推理評估上驗證。

## 面試題 {#interview-questions}

### Q：擴散語言模型和自迴歸模型有什麼不同，而那在哪裡是一種優勢？

**好的回答：**
自迴歸模型由左到右生成，每次前向傳播一個 token，每個都只以過去為條件。擴散語言模型則從一個完全遮罩的序列出發，在少數幾步內以雙向注意力對所有位置平行地反覆去噪。實務上的優勢是吞吐量：它每次前向傳播提交許多 token 而非一個，因此回報的速度跑到超過每秒 1,000 個 token，比經過速度優化的 AR 快上數倍。雙向性在填空與編輯上也有幫助，你遮罩數個位置並一次連貫地重新生成它們，以及在對順序不敏感的任務上。代價是在知識與困難推理上的品質差距，部分原因是遮罩擴散優化的是概似度下界而非精確概似度，加上較弱的長上下文與工具使用行為。所以我會在延遲關鍵、短到中等、結構化的生成（例如程式碼）上選用它，而不是把它當成前沿模型的替代品。

### Q：如果擴散 LLM 更快，為什麼大家還沒全部轉過去？

**好的回答：**
因為速度伴隨著實在的取捨，而且生態系還很年輕。品質差距在知識與困難推理上仍然顯著，而速度本身是「你解遮罩得多激進」的函數：最佳品質來自每步提交較少的 token，而這會把大部分速度還回去。純擴散在長上下文與可變長度輸出上也處理得很差，且缺乏 AR 已累積的成熟 KV cache 與服務優化。可能的路徑並非全面轉換，而是混合做法，區塊或半自迴歸模型以及「擴散作為推測式草稿產生器」，它們找回了快取與品質，同時保住大部分的吞吐量。在 2026 年，務實的做法是在延遲主導且輸出結構化的場景試行擴散，並把 AR 前沿模型留給其他所有情況。
