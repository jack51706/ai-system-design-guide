# 基準測試與排行榜

公開基準測試是這個領域談論模型能力的方式：MMLU、SWE-bench、GPQA、Arena Elo。它們對於建立方向感很有用，但用來做決策卻很危險。本章會教你每個主要基準測試在衡量什麼、哪些仍然能區隔前沿模型而哪些已經飽和，以及如何批判性地解讀一項基準測試的宣稱，這樣你才不會被一個數字騙倒。

承載全章的核心觀念：**一項基準測試的定義與已知缺陷是穩定的；它的分數則會過期。**排行榜數字每週都在變動，會被汙染與有利的測試框架（harness）灌水，而且越來越被捏造的條目所汙染。所以本頁會先講每項基準測試「衡量」什麼以及它如何「失效」，並把特定分數當成有日期、有出處、你應該重新查證的快照來看待。若要評估你自己的系統（那才是真正能預測生產環境品質的東西），請參閱 [LLM 評估](01-llm-evaluation.md)。

## 如何閱讀本頁 {#how-to-read-this-page}

下方每項基準測試都會列出它**衡量什麼**、它的**格式**、它的**飽和狀態**（前沿群聚是否高到分數差距已經是雜訊？），以及它的**已知缺陷**。在當前分數有助於建立方向感的地方，會附上出處與一個「待查證」標記，因為：

- **供應商自行回報的數字會高於獨立排行榜。**一間實驗室會用它能找到的最佳測試框架、最佳努力設定（best effort setting）與不設上限的基礎設施來回報自家模型。那個數字無法和另一間實驗室的數字相比，也無法和獨立執行的結果相比。只能比較由「同一個」測試框架產出的數字。
- **2026 年的網路被捏造的排行榜頁面汙染**，這些頁面會發明出模型與分數。如果你看到某個前沿程式設計分數被歸給一個你不認得的模型名稱，在主要來源（實驗室的貼文、基準測試自己的排行榜）確認之前，就假設它是 SEO 垃圾內容。
- **頭條數字隱藏了測試框架。**「SWE-bench 上 88%」在沒有代理鷹架（agent scaffold）、工具存取權、努力等級與輸出 token 上限的情況下是無法解讀的。

所以：用「衡量什麼」與「缺陷」這兩欄來推理一項基準測試，並把任何單一百分比當成有日期的資料點，而不是事實。

## 能力地圖 {#the-capability-map}

基準測試會依照它們所探測的能力來分組。在每一組之內，這個領域會持續淘汰已飽和的基準測試，並用更難的後繼者取而代之，所以實際的區隔指標每年都在移動。

### 一般知識與語言 {#general-knowledge-and-language}

大多已**飽和**：前沿模型群聚在 88-90% 以上，所以差距是雜訊。把這些當成歷史基線，而不是前沿的鑑別指標。

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **MMLU** | 57 個科目的多選學術知識（約 15.9k 題） | 已飽和（約 90%+ 群聚） | 歷史上最常被引用的基準測試。約 6.5% 的題目有標籤錯誤（依 MMLU-Redux）；網路汙染嚴重。已逐步從模型卡（model card）中淘汰。 |
| **MMLU-Pro** | 更難的 MMLU 後繼者：10 個選項而非 4 個、推理量大、約 12k 題 | 接近飽和 | 為了修正 MMLU 飽和而建立；現在撞到同樣的天花板（頂端群聚約 88-90%）。已因飽和而從 Artificial Analysis 指數中移除。 |
| **MMLU-Redux** | 對 MMLU 的錯誤修正重新標註（約 3k 題） | 已飽和 | 它存在的目的是「量化」MMLU 的錯誤率（發現 6.49% 錯誤，某些子集合高達 57%），而非排名前沿模型。 |
| **HellaSwag** | 常識性句子補完 | 完全飽和（95%+） | 人類基線約 95%；自 GPT-4 時代起即已解決。 |
| **ARC-Challenge** | 小學自然科學多選題 | 已飽和（約 96%+） | 實質上已解決。 |
| **WinoGrande** | Winograd 結構的指代消解／常識 | 已飽和（約 90%+） | 殘留的標註瑕疵。 |
| **BBH (BIG-Bench Hard)** | BIG-Bench 中最難的 23 項任務、多步推理 | 搭配 CoT 後飽和（>90%） | 已被 **BIG-Bench Extra Hard (BBEH)** 取代，後者正是因為 BBH 飽和而建立。 |
| **GLUE / SuperGLUE** | 較舊的 NLU 任務套件 | 已退役 | 模型在 2021 年初於 SuperGLUE 上超越人類基線。 |

### 前沿推理 {#frontier-reasoning}

這些是仍能區隔該領域頂端的基準測試，大致依剩餘餘裕（headroom）排序。

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **HLE (Humanity's Last Exam)** | 涵蓋 100 多個科目的封閉式專家題目、抗檢索（約 2,500 題，約 10% 為多模態） | 活躍、區隔幅度大 | 首屈一指的前沿知識基準測試。無工具分數大致落在 40-53% 之間，所以差距是可讀的。在 2025 年 1 月推出時，SOTA 還不到 10%，可見前沿移動之快。要留意有工具與無工具的設定，這會灌水並混淆比較。 |
| **GPQA-Diamond** | 198 道博士等級的生物／物理／化學題目、防 Google | 正朝飽和壓縮 | 設計上讓有網路存取的熟練非專家得分約 34%、博士專家約 65-70%。前沿群聚現在約 92-94%，所以它作為頂端鑑別指標正在變薄，但仍能區隔中段模型，是不錯的健全性檢查（sanity check）。 |
| **ARC-AGI-2** | 抽象格點謎題推理、考量效率（約 400 項任務）；抗記憶 | 活躍、強力區隔 | **請仔細閱讀：**人類小組通過 85%，而 85% 是大獎的「門檻」。彙整網站回報前沿模型「達到 85%」，但那是在非 ARC-Prize 測試框架、搭配大量測試時運算（test-time compute）下自行回報的數字。經 ARC-Prize 驗證的上限低得多（在高成本下約落在 50 多的中段）。絕對不要把門檻當成已達成的分數來引用。 |
| **FrontierMath** | 研究等級數學，是專家數學家要花數小時到數天才能解決的問題（約 350 題、私有；Tier 4 最難） | 活躍但快速攀升 | 開放式但有可自動驗證的封閉形式答案（防猜）。v1 約 42% 的題目有錯誤（已在 2026 年 6 月的 v2 修正）；它部分由 OpenAI 資助，所以單一實驗室的數字值得加上治理（governance）警語。Epoch 自己的說法是低於 70% 才是可達範圍，所以對遠高於此的彙整分數要存疑。 |
| **CritPt** | 由 50 多位物理學家設計、橫跨 11 個領域的研究級物理推理 | 活躍、天花板非常低 | 多數前沿模型在完整挑戰上的得分介於個位數到約 45%，所以區隔得很乾淨。較新（2025 年底），正成為標準的科學推理組成項目。 |

### 數學 {#mathematics}

經典的數學基準測試已被解決；活躍的區隔指標是最新的競賽年度與研究等級題組。

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **GSM8K** | 小學文字題 | 已飽和（>95%） | GSM-Symbolic 顯示當數字／子句被擾動時分數會下降，這證據顯示高分有一部分是記憶。 |
| **MATH / MATH-500** | 競賽數學（AMC/AIME 等級） | 已飽和（頂端約 99%） | MATH-500 是 500 題的子集合；規模小意味著頂端變異高。 |
| **AIME (2024/2025/2026)** | 奧林匹亞短答、整數答案、每年 30 題 | 第 N 年公開後即飽和 | 受重視是因為每一年都是新鮮的（低汙染），直到釋出為止。只有 30 題，所以單一年度分數變異高；偏好用多次平均的結果。2024/2025 已飽和。 |
| **HMMT, Putnam** | 更難的奧林匹亞／大學等級證明競賽 | Putnam 證明尚未飽和 | 證明評分依賴 LLM 裁判；只看最終答案的捷徑會高估真正的證明能力。 |

### 程式設計 {#coding}

片段式（snippet）基準測試用來排名已經沒用了；代理式、儲存庫規模的基準測試才是生產環境的訊號。

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **HumanEval / HumanEval+ / MBPP / MBPP+** | 從 docstring 合成單一函式，以單元測試檢查 | 已飽和（前沿約 90-97%） | 規模小且自 2021 年起公開，所以被汙染。HumanEval+ 增加約 80 倍的測試以抓出脆弱的程式碼；在 + 版本上分數下降代表薄弱的測試先前讓有 bug 的程式碼通過了。不要用來排名前沿模型。 |
| **SWE-bench Verified** | 解決真實的 GitHub issue 使隱藏測試通過（人工驗證過的 500 題子集合） | 接近飽和、被汙染 | 2024-2026 年程式設計的標準數字。但 OpenAI 自己的評估團隊指出超過 60% 的任務有瑕疵的測試，而且解法可以從 task ID 逐字重現。當成「約 80% 以上即前沿」的層級篩選器有用，但對精細排名不可靠。光是測試框架的選擇就會讓結果擺動 10-20 分。 |
| **SWE-bench Pro** | 更難、抗汙染的 issue 解決（私有＋ copyleft ＋保留的儲存庫） | 尚未飽和、正在浮現的主要訊號 | 分數比 Verified 低 25-35 分，且區隔好得多。OpenAI 現在在程式設計能力上指向 Pro 而非 Verified。仍要看測試框架：供應商回報的數字可能比同一模型在標準化測試框架上的數字高約 17 分。 |
| **SWE-bench Multimodal** | 解決視覺／UI 軟體問題（JS 前端、含損壞 UI 的截圖） | 尚未飽和 | 測試真正的視覺接地（visual grounding）；純文字版 SWE-bench 的系統會吃力。 |
| **SWE-rebench / SWE-bench-Live** | 持續更新、去汙染的 issue 解決；任務日期晚於模型的截止日 | 尚未飽和 | 最強的汙染稽核。SWE-rebench 特別能抓出在新鮮任務上崩潰的灌水 Verified 分數。 |
| **LiveCodeBench** | 依釋出日期標記的競技程式設計題目；只在截止日後的題目上計分 | 在新鮮時間窗內尚未飽和 | 在設計上抗汙染。競技程式設計不是軟體工程，所以它預測的是演算法推理，而非儲存庫工作。 |
| **Aider Polyglot** | 橫跨 6 種語言的真實多檔編輯，必須輸出有效的編輯格式，2 次嘗試 | 接近飽和（約 88%） | 衡量的是編輯格式的可靠度，而非儲存庫規模的推理；Exercism 來源是公開的（汙染）。 |
| **SciCode** | 拆解成子問題的研究級科學程式設計 | 遠未飽和 | 主流中最難的程式設計基準測試。注意有兩種互不相容的計分慣例（主問題對子問題）；務必註明引用的是哪一種。 |
| **Terminal-Bench (2.x)** | 在真實終端沙箱中的端對端代理任務（建置、除錯、系統管理） | 活躍、尚未飽和 | 分數與測試框架配對（代理 CLI 和模型一樣重要）。 |

**為什麼代理式基準測試取代了 HumanEval：**HumanEval/MBPP 對每個前沿模型都飽和在 90%+、規模小且已被記憶，而且單一函式從 docstring 到程式碼的任務無法預測真實工程能力。證據是：SWE-bench 在那些把 HumanEval 拿高分的同一個模型世代上，推出時的解決率不到 2%。閱讀儲存庫、執行測試、讀堆疊追蹤（stack trace）並反覆迭代修補（patch），是一種不同的技能，所以這個領域轉向了 issue 解決與終端基準測試。

### 代理式與工具使用 {#agentic-and-tool-use}

移動最快的領域，因為代理正是 2026 年生產環境價值所在。其中大多尚未飽和。

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **BFCL (Berkeley Function Calling Leaderboard)** | 工具／函式呼叫準確度；V4 加入搭配網路搜尋與記憶的代理式任務 | V1/V2 已飽和、V3/V4 未飽和 | 以 AST 為基礎的計分可能漏掉語意錯誤。當心排行榜的過時鏡像；請使用官方的 Berkeley 排行榜。 |
| **tau-bench / tau2-bench (Sierra)** | 代理在某個領域政策（零售、航空、電信）下使用工具並與模擬使用者對話 | 在可靠度上尚未飽和 | 回報 **pass^k**（把同一任務重跑 k 次），這暴露了殘酷的可靠度斷崖：一個 pass@1 約 60% 的代理可能掉到 pass^8 約 25%。這是與生產環境最相關的工具使用訊號，因為它衡量的是一致性，而不只是最佳情況。原始的 tau2 有 bug 任務；請使用修正過的分支（fork）。 |
| **GAIA** | 需要多步工具使用、網路瀏覽、檔案的通用助理任務 | 由編排的整體模型（ensemble）在頂端飽和（約 92%） | 頂端條目是多模型整體與鷹架，而非基礎模型，所以它衡量的比較是編排而非模型本身。 |
| **OSWorld-Verified** | 在真實作業系統上的電腦使用代理，以執行結果計分 | 尚未飽和、正逼近人類區間 | OSWorld 的清理與標準化後繼者（OSWorld 推出時最佳分數約 12% 對上人類約 72%）。供應商數字超過公開測試框架的數字。 |
| **Online-Mind2Web / WebArena** | 即時網路代理任務 | 混合 | Online-Mind2Web 的「進步的幻象」發現：許多商業代理在被透明地評判後，表現不如一個 2024 年的學術基線。裁判方法差異極大，所以分數往往無法比較。 |
| **GDPval** | 橫跨 44 種職業的真實具經濟價值的知識工作，由人類專家評分 | 活躍、尚未飽和 | 前瞻性的「它能不能做完一天真實工作」訊號；前沿以約 100 倍更低的成本逼近專家交付物品質。成對 Elo 的變體（GDPval-AA）與勝率版本不同。 |
| **METR time-horizon** | 模型在 50% 可靠度下能完成的任務長度（以人類分鐘計） | 活躍的研究標準 | 不是排行榜而是一個趨勢：整體而言時間範圍（horizon）大約每 7 個月翻倍，在程式設計上更快。是談論代理自主性成長最乾淨的方式。 |

### 長上下文 {#long-context}

頭條發現：**廣告宣稱的上下文視窗高估了可用的上下文。**先講這一點，而不是先講規格表上的視窗大小。

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **NIAH (needle-in-a-haystack)** | 在不同深度與長度下的單一事實檢索 | 已飽和／微不足道 | 前沿模型得分約 100%，這給人虛假的信心。是健全性檢查，不是鑑別指標。 |
| **RULER (NVIDIA)** | 在受控長度下的檢索、多跳追蹤、聚合、QA；回報「有效長度」 | 尚未飽和 | 「有效長度對廣告長度」落差的大本營：許多宣稱 128K 的模型只在約 32-64K（廣告值的 25-50%）內維持品質。是合成的，所以要搭配以散文為基礎的測試。 |
| **Fiction.LiveBench** | 最高約 192K token 的深度敘事理解（心智理論、時序、隱含推論） | 尚未飽和 | 最嚴苛的實務長上下文測試；多數模型到 192K 時跌破 80%。規模很小（36 題），所以有雜訊。 |
| **MRCR (multi-round coreference)** | 在多個近乎相同的針之間分辨，並回傳第 i 根 | 尚未飽和，尤其是 8 根針 | 分數隨針數與長度急遽崩潰。 |
| **LongBench v2 / LongBench Pro** | 8K-2M token 的真實長上下文理解與推理 | 尚未飽和 | LongBench Pro 的發現：長上下文「優化」勝過單純的參數量擴展，而且每個模型的有效長度都低於廣告值。 |

一句可放進設計文件、站得住腳的話：*廣告宣稱的上下文視窗經常高估可用上下文；在 RULER 上，許多宣稱 128K 的模型只在約 32-64K 內維持品質，不過前沿正在快速進步。*「上下文腐化」（context rot）的研究（劣化在遠未到視窗上限前就開始，有時甚至出現一個反直覺的結果：打亂順序的乾草堆勝過連貫的文件）更強化了要以較小的有效預算來設計。請參閱 [上下文工程](../05-prompting-and-context/05-context-engineering.md)。

### 多模態 {#multimodal}

影像多選基準測試正在飽和；影片推理才是真正尚未解決的前沿。

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **MMMU / MMMU-Pro** | 橫跨 6 個學科的大學等級多模態推理 | 接近飽和 | MMMU-Pro 加強了難度（10 個選項、純視覺題目，問題就在截圖裡），但有嚴重的跨測試框架不一致（同一個模型在不同測試框架上被回報為 81% 與 94%），所以絕對不要跨來源比較。 |
| **MathVista, DocVQA, ChartQA, MMBench** | 視覺數學、文件 QA、圖表 QA、廣泛多模態 | 在前沿大致飽和 | DocVQA/ChartQA 近乎解決；寬鬆準確度（relaxed-accuracy）計分會隱藏數值錯誤。 |
| **Video-MME / Video-MME-v2** | 影片理解 | v1 接近飽和、v2 遠未飽和 | Video-MME-v2（2026）採用非線性分組計分，顯示出很大的模型對人類落差，所以它是活躍的多模態區隔指標。 |

### 事實性與指令遵循 {#factuality-and-instruction-following}

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **IFEval** | 可程式化驗證的指令遵循（「至少 400 字」、有效 JSON、不含逗號） | 大致飽和（約 90%+） | 只檢查「可檢查的」約束，是指令遵循中很窄的一部分；可被取巧。 |
| **SimpleQA / SimpleQA Verified** | 短答、閉卷的事實回憶、對抗式地難；獎勵經校準的棄答（abstention） | 尚未飽和 | F1 上限約在 50 多的中段，所以幻覺遠未被解決。頭條教訓：更有能力不代表更符合事實。 |
| **TruthfulQA** | 抵抗常見的迷思誤解 | 老化／部分飽和 | 靜態且廣為人知，所以被汙染；「真相」標籤有爭議；可用模稜兩可的回答取巧。 |
| **FACTS Grounding** | 長篇答案是否完全由提供的來源所支持（沒有未接地的宣稱） | 尚未飽和（頂端約 0.88） | 由一組與參賽者有共同血緣的 LLM 整體來評判；條目是自行回報的。是用於 RAG 忠實度（faithfulness）的正確基準測試家族。 |

### 人類偏好 {#human-preference}

| 基準測試 | 衡量什麼 | 狀態 | 備註 |
|-----------|----------|--------|-------|
| **LMArena (Chatbot Arena Elo)** | 群眾外包的盲測成對偏好，以 Elo 回報 | 沒有天花板，但頂端約 15 名壓縮在約 25 個 Elo 點之內 | 著名的偏好訊號，也是最常被誤用的。請參閱 [排行榜幻象](#the-leaderboard-illusion)。一律閱讀**風格控制過（style-controlled）**的 Elo（它會迴歸掉長度與格式偏誤）以及信賴區間；約 15-20 Elo 以內的差異是雜訊。 |
| **Arena-Hard-Auto v2** | Arena 的自動、可重現替代指標：由強力 LLM 成對評判的困難提示 | 尚未飽和、強力區隔 | 區隔力約為 MT-Bench 的 3 倍，與人類 Arena 排名約 98% 相關。當你需要時，是便宜又可重跑的偏好訊號。使用時開啟風格控制。 |
| **MT-Bench** | 透過 LLM 裁判評估多輪品質（80 個提示） | 已飽和／過時 | 規模小；有 GPT-4 裁判偏誤（偏好自己、偏好冗長）。已被 Arena-Hard 取代。 |

## 批判性地解讀基準測試 {#reading-benchmarks-critically}

這才是真正重要的部分。任何人都會讀排行榜；「正確地」讀它才是技能。

### 飽和 {#saturation}

當前沿群聚得太靠近天花板，以致分數差距落在雜訊之內時，一項基準測試就飽和了。MMLU 是典型案例：GPT-4 在 2023 年初達到約 86%，而前沿自此就停在 86-93%，所以 2 分的「勝出」往往只是提示的產物（MMLU 分數在不同提示措辭下會變動 4-5%）。可用的判準是：**當領先者群聚在約 3 分之內，排名次序就是統計雜訊，而非能力。**

當一項基準測試飽和，這個領域的回應方式是：(1) 更難的後繼者（MMLU 然後 MMLU-Pro 然後 HLE），(2) 私有或保留的題組，(3) 時間閘控（time-gated）的「即時」基準測試，以及 (4) 複合指數。一項靜態公開基準測試的中位有用壽命不到約 2 年。

### 汙染 {#contamination}

基準測試是公開的，會被爬取進預訓練資料，所以模型可以靠記憶而非能力拿高分。證據很直接：重新衍生 HumanEval 風格的題目（EvoEval）讓 51 個模型的分數下降約 39%；在 LiveCodeBench 上，某個模型在其截止日之前的題目上 pass 率約 60%，截止日之後卻掉到約 0%；OpenAI 發現 SWE-bench Verified 的解法可以從 task ID 逐字重現。橫跨多個多選 QA 基準測試，測得的汙染範圍從 1% 到 45%，而且較大的模型從中獲益更多。

抗汙染的設計：**時間閘控**（只在模型截止日之後釋出的題目上計分，如 LiveCodeBench 與 SWE-rebench 所做），**私有保留題組**（FrontierMath、ARC-AGI-2；代價是無法重現），以及**金絲雀字串**（canary string，植入資料集中的一個獨特 token，如果模型背出它就標記為汙染）。偵測方法（n-gram 重疊、成員推論、TS-Guessing 測驗）都有失效模式；尤其成員推論攻擊在真實的預訓練模型上幾乎只比隨機猜好一點。實務做法：對任何靜態公開基準測試，假設有某種程度的汙染，並對絕對數字打折扣。

### 測試框架與鷹架的變異 {#harness-and-scaffold-variance}

同樣的模型權重，依提示、是否有工具可用、推理努力等級與代理鷹架的不同，分數會差 10-20 分。Anthropic 量測到光是基礎設施組態「本身」（RAM、並行度，甚至一天中的 API 延遲時段）就讓 Terminal-Bench 結果移動約 6 分。這就是為什麼**供應商自行回報的數字會高於獨立排行榜**：實驗室會在不設上限的基礎設施上，回報它們為自家模型找到的最佳測試框架與努力等級。隨之而來的硬性規則：**絕對不要拿一個供應商的數字去比另一個供應商的數字，或去比獨立排行榜。**只有同一測試框架的數字才能比較。而且推理努力不是單調的，在一項大型代理研究中，36 種設定裡有 21 種是「想更多」反而降低了準確度，所以「高努力」的供應商數字甚至無法和同一模型的預設努力獨立執行結果相比。

### 排行榜幻象 {#the-leaderboard-illusion}

對 LMArena 的核心批判（Cohere 等人，稽核約 200 萬場對戰、243 個模型）發現四個問題：供應商私下測試許多變體卻只發布最佳的（Meta 在 Llama-4 之前測試了 27 個變體），這違反了 Elo 數學背後無偏抽樣的假設；專有供應商取得的對戰資料遠多於開放模型；你可以「針對」Arena 分佈訓練以取得大幅勝率提升；以及被悄悄棄用的模型扭曲了排名。LMArena 的反駁對幅度有異議（他們估計私下測試的提升約為 11 Elo，並隨新票數累積而衰減），並指出過度擬合的數字是在一個靜態替代指標上量測的，而非即時的人類排行榜。請把這呈現為**有爭議但有實證支持。**

無論如何，實務指引都一樣：把 Arena Elo 當成**一般聊天偏好的衡量，而非正確性、事實性或硬推理的衡量**；一律使用風格控制過的排行榜（Arena 獎勵更長、更漂亮的答案）；閱讀信賴區間（頂端約 15 名在統計上幾乎並列）；並把它當成三種訊號之一，絕不單獨使用。

### 基準測試到生產環境的落差 {#the-benchmark-to-production-gap}

一個公開分數要能預測你的生產環境表現，只有當三個條件同時成立時才行：基準測試所測的任務與你的相似、測試集乾淨無汙染，而且基準測試尚未飽和。實務上這三者很少同時成立。一項對基準測試分數的主成分分析發現，單一的「一般能力」因素只能解釋約 50% 的變異；其餘是模型家族的特異性與雜訊，這就是為什麼兩個一般能力相當的模型，在「你的」任務上可能差很多。高 GPQA 並不保證在你的領域上的表現。

對程式設計與代理而言，最好的公開替代指標是 GPQA-Diamond 與 SWE-bench Verified（Aider Polyglot 與 AIME 風格的題組在一般能力上也乾淨地載入），但只在測試框架相符時才成立。每位實務工作者都會得到的結論：**對你的決策而言，忽略排行榜，在你自己的資料上建立評估。**建構一個橫跨功能、情境與人物誌（persona）切分的黃金集（gold set）；使用一個對領域專家校準過的二元 LLM-as-judge（以精確率和召回率衡量，而非原始一致率）；並以成本來定價能力，因為沒有任何公開基準測試含有成本訊號。請參閱 [LLM 評估](01-llm-evaluation.md) 以及 [白板演練](../00-interview-prep/04-whiteboard-exercises.md) 中的評估管線白板習題。

### 複合指數 {#composite-indices}

因為任何單一基準測試都會在一兩年內飽和，這個領域以加權複合指數來排名前沿模型，這些指數會在組成項目達到上限時持續鑑別，並抵抗對單一測試的過度擬合：

- **Artificial Analysis Intelligence Index** 對橫跨代理、程式設計、科學推理與一般知識的約 9 項評估加權，並在組成項目飽和時改版（它已隨時間移除 MMLU-Pro、LiveCodeBench 與 AIME 2025）。
- **Epoch Capability Index (ECI)** 在約 1,000 多項評估上擬合一個項目反應理論（item-response-theory）模型，以統計方式推論每項基準測試的難度，使模型在更「難」的基準測試上表現好時得分更高。
- **HAL (Holistic Agent Leaderboard, Princeton)** 是專為代理設計、考量成本的複合指數：它同時計分準確度「與」金錢成本、在各模型上執行固定的測試框架，並用日誌分析揭露走捷徑的代理（從 arXiv 拉答案而非真正解題）與評估 bug。它之所以存在，是因為在 10 倍成本下換來 1% 的準確度提升並不算勝出。

複合指數是回答「哪個模型整體最好」的正確工具，但它們仍會繼承組成項目的缺陷，所以要讀它們聚合了什麼。

## 一份實務檢查清單 {#a-practical-checklist}

當你閱讀任何基準測試宣稱時：

1. **讀測試框架，而不只是數字。**要求提供鷹架、工具存取權、努力等級與輸出 token 上限。一個光禿禿的百分比無法解讀。
2. **檢查信賴區間。**忽略雜訊帶之內的差距。SWE-bench Verified 只有 500 題（每題約 0.2%）；Arena 約 15-20 Elo 以內的差距是雜訊。在組態相符之前，不要相信任何約 3 分以內的差距。
3. **偏好時間閘控、私有或保留的題組。**對靜態公開題組（MMLU、HumanEval、GSM8K），假設有汙染並對絕對數字打折扣。
4. **絕不跨測試框架或跨自行回報比較。**只有同一測試框架的數字才能比較。
5. **健全性檢查模型名稱。**如果一個前沿分數被歸給一個你找不到主要來源的模型，它很可能是捏造的。已確認的 2026 年 6 月前沿家族：Claude（Fable 5、Opus 4.8/4.7、Sonnet 4.6、Haiku 4.5）、GPT-5.5、Gemini 3.1 Pro、DeepSeek V4、Llama 4、Kimi K2.6、Qwen 3.6、Mistral Medium 3.5、Grok 4.3。
6. **在相信一個排名之前，三角驗證三種訊號類型**：一個靜態學術評估、一個人類偏好競技場、一個代理式套件。
7. **對你的決策，在你自己的資料上建立評估。**公開分數很少能預測你的領域。排行榜告訴你要把誰列入候選；你的黃金集告訴你要出貨給誰。

## 2026 年哪些基準測試重要 {#which-benchmarks-matter-in-2026}

要在**前沿一般能力**上建立方向感：HLE、GPQA-Diamond（但要注意它正在變薄）、CritPt，以及複合指數（Artificial Analysis Intelligence Index、Epoch ECI）。忽略 MMLU、MMLU-Pro、HellaSwag、ARC-Challenge。

對**程式設計與代理**：SWE-bench Verified 當作層級篩選器、SWE-bench Pro 與抗汙染的即時變體（SWE-rebench、SWE-bench-Live）用於真實排名、Terminal-Bench 與 tau2-bench 用於工具使用與可靠度（留意 pass^k），以及 HAL 用於考量成本的代理比較。忽略 HumanEval/MBPP。

對**長上下文**：RULER 與 Fiction.LiveBench 優於 NIAH；以有效上下文而非廣告上下文來設計。

對**事實性**：SimpleQA 用於閉卷幻覺、FACTS Grounding 用於 RAG 忠實度；記住更有能力不代表更符合事實。

對**偏好**：搭配信賴區間的風格控制過 Arena Elo，或 Arena-Hard-Auto v2 作為可重現的替代指標。

而對**你的產品**：以上皆非。建立你自己的黃金集與裁判。這方面的前沿研究移動很快（代理可靠度科學、宣稱層級的忠實度評估、模型偵測自己正在被測試的評估覺察 eval-awareness）；請參閱 [研究雷達](../RESEARCH-RADAR.md)。

## 面試問題 {#interview-questions}

### Q：某供應商說他們的模型在 SWE-bench Verified 上得分 90%。在相信它能預測你的程式設計代理品質之前，你會問哪些問題？

**強答案：**
首先是測試框架：哪個代理鷹架、什麼工具、什麼努力等級、什麼輸出 token 上限？相同的權重光是鷹架就會擺動 10-20 分，而供應商數字用的是他們找到的最佳測試框架，所以那 90% 無法和任何其他模型已發布的數字相比。其次是汙染：SWE-bench Verified 部分被汙染（OpenAI 發現解法可從 task ID 重現），所以我會想要抗汙染的變體（SWE-bench Pro、SWE-rebench），那裡的分數低 25-35 分且區隔更好。第三是信賴區間：它是 500 題，所以幾分是雜訊。第四也是最重要的，生產環境落差：即使是乾淨的 SWE-bench 數字，也只有當我的任務類似 GitHub issue 解決時才能預測我的程式碼庫。我會用公開數字列出候選，再用我自己那組來自我們儲存庫的真實工單黃金集（由校準過的裁判計分、並以成本定價）來做決定。

### Q：為什麼像 MMLU 與 HumanEval 這樣的基準測試不再適合用來排名前沿模型，又有什麼取代了它們？

**強答案：**
它們飽和了：前沿在 MMLU 上群聚於 88-90% 以上、在 HumanEval 上 90%+，所以差距落在提示措辭的雜訊之內。它們也規模小且公開，所以被汙染，重新衍生 HumanEval 題目會讓分數下降約 39%。而且構念太簡單：單一函式從 docstring 到程式碼的任務無法預測真實工程能力，這就是為什麼 SWE-bench 在那些把 HumanEval 拿高分的同一批模型上，推出時不到 2%。這個領域用以下取代了它們：更難的後繼者（MMLU 到 MMLU-Pro 到 HLE）、抗汙染的時間閘控基準測試（LiveCodeBench、SWE-rebench）、代理式與儲存庫規模的基準測試（SWE-bench、Terminal-Bench、tau2-bench），以及在組成項目飽和時持續鑑別的複合指數。

### Q：在為一個聊天產品選擇模型時，你會如何負責任地使用 LMArena Elo？

**強答案：**
當成一般聊天偏好的一個訊號，絕不當成正確性或推理的衡量。我會閱讀風格控制過的排行榜，因為原始 Arena 不管正確與否都獎勵更長、格式更好的答案；我也會閱讀信賴區間，因為前十幾個模型在統計上於約 15-20 Elo 之內幾乎並列。我也會因為排行榜幻象效應而對它打折扣：供應商私下測試許多變體並只發布最佳的，所以一個新進的頂端條目可能部分是 best-of-N 的運氣。然後我會用一個客觀基準測試與一個代理式套件來三角驗證，最終在我自己的偏好資料上驗證，因為 Arena 的提示不是我使用者的提示。
