# 多模態生成

生成圖像、影片與音訊，和理解它們是截然不同的工程問題。一個生成式媒體產品是一條**由長時間執行、非確定性、昂貴的工作所組成的資產處理管線**，而大部分工作就在於安全地編排這些工作、證明來源出處，並評估你無法精確重現的品質。本章以這些長期不變的考量為主軸，把特定的模型當成易腐的快照來看待。除非有連結到第一手來源，否則下方每一項能力數據都是某方回報或廠商宣稱的數字。

**範圍界定（請先讀）：** 這個主題對創意、媒體、行銷、遊戲與虛擬人偶（avatar）產品至關重要，但對後端服務、資料平台、RAG 與純文字代理而言大多是雜訊。唯一會觸及到*任何*涉及使用者上傳或 AI 生成媒體之產品的部分，是**來源出處與安全**，因為法律義務是依附在生成或散布合成媒體這件事上，而非你的領域。如果你的系統永遠不會產出任何一個像素或一段波形，那麼就只有那一節適用於你。

## 生產環境管線模式

這是比任何模型都更為長久的部分。

**串接是一張 DAG，而不是單次呼叫。** 標準的創意管線是一張由多個階段組成的圖，每個階段往往是來自不同廠商的不同模型：`prompt -> image (keyframes/style) -> image-to-video -> lip-sync -> voice/TTS -> music/SFX -> mux`。長期不變的教訓是**在階段邊界上做關注點分離（separation of concerns）**，如此一來任何階段都可以替換，這與 [voice agents](../18-voice-and-audio-agents/01-realtime-voice-agents.md) 一章所描繪的、相同的串接式（cascaded）對比單體式（monolithic）取捨是一樣的。節點圖（node-graph）工具（ComfyUI）為圖像與影片把這件事編成了規範；請把工作流程圖當成有版本控管的程式碼，而不是 UI 狀態。

**非同步、佇列與長工作處理是必備的。** 圖像生成只要數秒；**影片生成則是每段片段數十秒到數分鐘**，因此沒有任何同步請求撐得過它。通用的模式是：提交時立即回傳一個工作 ID，工作在 GPU 工作節點池（worker pool）上執行，而客戶端透過 webhook 回呼（callback）得知工作完成（並附帶輪詢備援，因為 webhook 會被丟掉）。要驗證 webhook 簽章，並**依佇列深度（queue depth）自動擴展**，而不是依 CPU。由於單一影片工作會耗費真金白銀與數分鐘，請為每個邏輯請求附上一個冪等性鍵（idempotency key），這樣客戶端重試或 webhook 重送才不會付兩次錢。

**成本與延遲控制**，大致按槓桿效益由大到小排序：用完整的請求指紋（prompt 加上所有參數加上 seed 加上模型版本）做快取，讓相同的請求永遠不會重複計費；以低成本打草稿（低解析度、少步數、一個又小又快的模型），只對核准後的草稿用全品質重新算圖，這是影片中單項節省效益最大的做法；在支援的地方做批次處理；審慎挑選方案層級（turbo 與蒸餾過的變體以品質換取可觀的延遲與成本優勢）；並維持暖機的工作節點池，因為載入一個數 GB 的檢查點（checkpoint）可能要花數十秒。

**資產與提示管理。** 請把 prompt、負面提示（negative prompt）、seed、模型與版本，以及所有調節輸入（conditioning input，例如控制圖、參考圖、adapter ID 與權重）當成一筆結構化、有版本控管的紀錄，與每一份輸出一起儲存。沒有完整的參數集，你就無法重現或除錯一次生成，而且供應商會在不變的名稱背後悄悄更換模型。這份清單（manifest）同時也是你的來源出處紀錄。

**Seed 與可重現性，殘酷的真相。** seed 能讓一次*在本地、固定釘住*的執行可重現，但只要一跨越硬體或批次邊界，可重現性就會劣化，而在託管 API 上則實質上根本不存在。根本原因是通用性的，並非生成領域所特有：浮點運算不滿足結合律，因此不同的 GPU 核心（kernel）會以不同順序累加；批次大小會改變核心策略（這是數值雜訊最常見的來源）；而某些 GPU 運算除非被明確強制，否則是非確定性的。實務立場：自架（self-hosted）並釘住硬體、固定批次、固定函式庫版本，再加上一個 seed，可以得到近乎可重現的圖像；任何託管 API，尤其是影片，都應當視為不可重現，因此要圍繞感知相似度（perceptual similarity）來設計 QA，而非精確比對。

## 來源出處與安全

技術不斷翻新；但標示的義務，以及無法完美執行該義務這件事，是長久不變的。兩個互補的層級，**而且兩者都可被移除。**

**C2PA / Content Credentials** 是一套開放標準，以密碼學方式把來源出處綁定到資產上（「媒體的營養標示」）。一份清單（manifest）持有關於建立與編輯的斷言（assertion）、一份簽署過的主張（claim），以及對來源資產的參照，並具備**硬綁定**（位元組的密碼學雜湊，可偵測竄改）與**軟綁定**（能在重新編碼後存活的指紋或浮水印）。「Durable Content Credentials」會藉由在外部儲存庫中查找一個存活下來的浮水印或內容指紋，來復原被剝除的清單，它之所以存在，正是因為清單會因重新編碼與截圖而與資產分離。採用是真實且持續成長的：主要的圖像與影片生成器現在會附上 C2PA 中繼資料與浮水印，有些相機製造商會在拍攝當下為照片簽章，而平台會讀取憑證以套用 AI 標示。一個值得拿來教學的警世案例：某相機製造商在一個簽章金鑰漏洞之後，先加入又暫停了 C2PA，因為信任模型的可靠程度，就只等同於金鑰保管的可靠程度。

**浮水印（SynthID 式）** 為生成的媒體加上一個無法察覺的訊號。給工程師的長久警語：**隱形浮水印對一個有動機的對手而言並不穩固。** 經同儕審查的研究顯示，一種再生成攻擊（加入雜訊、再用擴散模型去雜訊）在擾動門檻以下可被證明能移除隱形圖像浮水印，同時保留品質，而文字浮水印則會被改寫（paraphrase）與回譯（back-translation）所削弱。浮水印能嚇阻隨意的濫用，並讓平台得以標示；但它們阻止不了對手。請把 C2PA 簽署過的來源出處、一個能在中繼資料被剝除後存活的浮水印，以及以分類器為基礎的偵測層層疊加，並假設每一層單獨都會被攻破。

**安全過濾器**會套用輸入審核（封鎖不被允許的 prompt）與輸出審核（NSFW 與身分分類器），然後才回傳資產。高嚴重性的類別包括未經同意的私密影像、真實人物的深偽（deepfake）、未經同意的語音複製，以及著作權或肖像權侵害；緩解措施包括 prompt 與地區封鎖清單、已知人臉拒絕、需經同意才能使用肖像、速率限制，以及與來源出處清單相繫的濫用記錄。

**2026 年的監管背景**（這是會觸及到那些僅僅託管媒體之非媒體產品的部分）：EU AI Act 的透明度條款要求供應商以機器可讀的方式標記生成的輸出，並要求部署者揭露深偽，相關義務自 2026 年 8 月起適用（見 [AI Governance and Compliance](../13-reliability-and-safety/04-ai-governance-and-compliance.md)）；美國的 TAKE IT DOWN Act（2025）針對包含 AI 深偽在內的未經同意私密影像，課予平台下架義務；而各州的肖像權法（田納西州的 ELVIS Act 與許多性露骨深偽法規）則保護聲音與肖像。音樂與語音生成另外還背負著尚未定論的著作權訴訟，這使得一個模型的授權與賠償保障（indemnity）狀態成為一個實實在在的採購問題。

## 評估生成品質

兩個無可迴避的事實：自動化指標很弱，而人類偏好才是真正的基準真相（ground truth），但很昂貴。

**自動化指標，以及它們為何不可靠。** FID 這個歷史悠久的圖像指標，會與人類評分者相牴觸、對失真處理不當，而且在實際的樣本量下既有偏差又不穩定；CLIPScore 量測文字與圖像的對齊，但可以被鑽空子，而把多個弱指標組合起來並不會變成一個強指標。FVD 繼承了 FID 的問題，並且在影片典型的小樣本量下並不穩定。對於音訊，FAD 是 FID 的對應物，帶著相同的警語，而 MOS（主觀 1 至 5 分）在 TTS 已逼近人類品質的現在，受限於天花板效應。請把這些當成粗略訊號使用，絕不要當成正確性的依據。

**人類偏好是黃金標準。** 這個領域以盲測的成對競技場（pairwise arena）為圖像與影片模型排名，並透過 Elo 或 TrueSkill 在大量投票上計分。請把公開競技場的排名當成粗略訊號，並在*你自己*的 prompt 分布上驗證，這與 [benchmarks](../14-evaluation-and-observability/03-benchmarks-and-leaderboards.md) 一章為 LLM 所開出的、相同的紀律是一樣的。

**為不可重現的管線做回歸測試**是長久不變的實務，因為供應商會在不變的名稱背後悄悄更新模型。建立一個有版本控管的黃金 prompt 集，涵蓋你真實的使用情境與失敗模式；在託管 API 上捨棄精確比對，改為比較指標分數的*分布*；使用感知相似度（感知雜湊、SSIM、與基準的 CLIP 嵌入距離）而非相等性；針對雜湊看不到的部分，加上一個依評分準則（prompt 遵循度、瑕疵、品牌安全）運作的 VLM-as-judge；長期追蹤評審與指標的分布並在漂移時告警，漂移往往意味著上游供應商更換了模型；並在 API 允許之處釘住有日期標記的模型版本。以統計顯著性而非原始差值來把關發布，並讓 prompt 獨立於模型進行版本控管。

## 模型版圖

保持輕量且標註日期，因為它每個月都在翻新，而且流通中的規格表細節有許多來自低品質的來源。

- **圖像。** 開放權重（open-weight）的前沿由 FLUX 系列領軍（近期的一個版本是一個大型的 rectified-flow transformer，經量化後可在單張高階消費級 GPU 上執行，而且值得注意的是，它在角色或風格參照上不需要微調）；Stable Diffusion 一脈則仍是廣泛的工具基礎。值得拿來教的授權陷阱：開放*權重*不等於開放*使用*，因為有好幾個變體（`dev` 系列）是非商業的，商業用途需要付費授權，而蒸餾過的 `schnell` 變體則採寬鬆授權（Apache-2.0）。調節（conditioning）可以組合：ControlNet（從姿勢、深度、邊緣或分割圖做結構控制）、參考 adapter（把一張圖當成風格或肖像的 prompt 來用）、修補（inpainting）與外擴（outpainting），以及區域提示（regional prompting）；生產環境的組合是在同一張圖裡放入一個身分 adapter 加上結構控制再加上一個文字主題。LoRA 是風格或主體個人化的預設做法。
- **影片。** 專有 API 的領頭者（OpenAI 的 Sora 2、Google 的 Veo（具備原生同步音訊），外加來自 ByteDance、Kuaishou、Runway 等的強力競爭者）坐落於一個快速進步的開放權重層之上。一個值得標示的更正，也是這一節衰退得有多快的活生生範例：OpenAI 在 2026 年 3 月棄用了 Sora 2 / Videos API（據報導將於 2026 年稍晚關閉），並在不久後停掉了消費端的 Sora app，所以在以某個影片模型為基礎開發之前，請先查證它的狀態，而不要相信任何快照，包含本章在內。計費是按輸出的每秒計算（大致從幾美分到不到一美元每秒），生成需要數十秒到數分鐘，而輸出不可重現，這迫使你採用非同步工作、先打草稿再算圖，以及對每位使用者設下硬性成本上限。
- **音訊。** 語音與 TTS（ElevenLabs 是標竿，具備複製與配音功能）與音樂（Suno 與 Udio，正處於持續進行中的著作權訴訟與授權和解的核心）。與影片的組合，呼應了 voice 一章的分歧：原生的聯合音訊與影片以最少的控制提供最佳的同步，而串接式做法（先生成無聲影片，再分別加上 TTS、音樂與 lip-sync）則以較高的工作量為代價，提供獨立的控制與配音能力，這正是為什麼大多數生產環境的配音管線維持串接式的原因。

## 面試問題

### Q：設計一個把腳本轉成有旁白、有配樂之影片的服務骨幹。難的部分是什麼？

**好的回答：**
這個骨幹是一張由多個階段組成的非同步 DAG，每個階段都是一個可替換的模型：prompt 到關鍵影格圖像、image-to-video 產生動態、TTS 做旁白、音樂模型做配樂，以及 lip-sync 把聲音綁到影片上，然後是一個 mux 步驟。由於影片生成需要數十秒到數分鐘，每一個階段都是一個工作：提交回傳一個 ID，工作在一個依佇列深度自動擴展的 GPU 工作節點池上執行，完成時觸發一個 webhook 並附帶輪詢備援。難的部分是成本與可靠性。成本：用完整的請求指紋做快取、以低成本打草稿且只對核准的草稿用全品質重新算圖，並對每位使用者的花費設上限，因為每一次重試都會耗費真金白銀。可靠性：附上冪等性鍵，讓被重試或重送的工作不會付兩次錢，並為每份資產儲存一份包含 prompt、seed、參數與模型版本的完整清單，既為了除錯，也作為來源出處紀錄。我還會用輸入與輸出審核來把關生成，並附上 C2PA 憑證，因為合成媒體的義務不論我的領域為何都適用。

### Q：當輸出不可重現時，你如何為一條生成式管線做回歸測試？

**好的回答：**
你放棄精確比對，改用感知與統計的方式來測試。我會維護一個有版本控管的黃金 prompt 集，涵蓋真實使用情境與已知的失敗模式，並在每次變更時執行它。在可重現性成立之處（自架並釘住硬體與 seed），我可以用 seed 鎖定的檢查；在託管 API 上我假設不可重現，因此我比較指標分數的分布而非單一輸出，並使用感知相似度、感知雜湊、SSIM，以及與基準的 CLIP 嵌入距離，再加上一個依評分準則運作、針對 prompt 遵循度與瑕疵的 VLM-as-judge。我長期追蹤那些分布並在漂移時告警，因為突然的偏移通常意味著供應商在不變的名稱背後悄悄更新了模型，所以我也會在 API 允許時釘住有日期標記的模型版本。發布以統計顯著性而非原始差值來把關，而且我讓 prompt 獨立於模型進行版本控管，這樣我才能把一次回歸歸因到正確的變更上。
