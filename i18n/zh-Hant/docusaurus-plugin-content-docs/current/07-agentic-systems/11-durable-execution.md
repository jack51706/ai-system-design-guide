# 長時間運行代理的持久化執行（Durable Execution）

一次代理運行並不是一個請求／回應的處理器。它會呼叫工具、讀取文件、觸發動作、等待核准，並在可能持續數分鐘、數小時或數天的多個步驟之間攜帶狀態。這與一般的基礎設施相衝突：行程會被殺掉、節點會被回收、部署會輪替 pod。一個天真地把狀態保存在記憶體中的代理迴圈，在上述任一事件發生時都會失去一切，而一個天真的重試又會重新執行副作用。**持久化執行（Durable Execution）** 就是讓長時間運行的代理能熬過這一切的紀律。本章涵蓋其模型、工具、它如何對應到代理迴圈，以及何時值得承擔這份複雜度。

## 為什麼代理打破了一般的故障模型

代理是長時間運行、有狀態、會產生副作用的行程，這打破了三項假設：

- **剛好一次（Exactly-once）的副作用。** 如果一次工具呼叫成功，但代理在記錄它之前就崩潰，那麼一個被恢復的運行可能會重試該呼叫，意味著重複的付款、工單或部署。核心的模糊性在於：在活動進行到一半崩潰之後，你無法判斷副作用究竟已經提交，還是只是它的確認訊息遺失了，因此天真的重試會把它送出兩次。
- **能熬過重啟的人類介入（human-in-the-loop）暫停。** 一個代理可能需要為了一項核准而阻塞數小時或一天，卻不能失去進度或燒掉運算資源。一個保存在行程記憶體中的暫停，會在下一次部署時消亡。
- **被記錄下來的非確定性。** 你無法重播一次 LLM 呼叫並假裝它是同一個事件；相同的提示可能產生不同的回應。其輸出必須在第一次就被記錄下來，並在恢復期間重複使用。

天真的重試會重新執行副作用；而天真的檢查點（checkpoint）只在步驟之間保存狀態，這仍然在「執行一個副作用」與「記錄其結果」之間留下了一個不安全的時間窗。要傳達的區別是：**檢查點捕捉的是狀態，但持久化執行捕捉的是步驟的日誌（journal）**，而你無法僅憑一份狀態快照就在副作用進行到一半時安全地恢復。

## 持久化執行模型

核心模式是 **以程式碼形式表達的工作流（workflows-as-code）加上一份僅可追加（append-only）的事件歷史，再加上確定性重播（deterministic replay）。** 像 Temporal 這樣的系統會為每個工作流記錄一份不可變的事件歷史；如果某個 worker 在 10 個步驟中的第 5 步崩潰，另一個 worker 會重播這份歷史，以重建記憶體內的狀態，並從第 6 步恢復。

確定性限制從何而來，正是這裡承重的概念：恢復是 *透過重播* 來運作的，所以在重播期間，執行中的步驟必須與日誌中的步驟一致，否則系統無法保證恢復成功。這意味著 **工作流程式碼本身必須是確定性的**：不能直接呼叫當前時間、不能有亂數或 UUID、不能有直接的網路呼叫、工作流程式碼內部不能有非確定性的執行緒交錯。非確定性與副作用都被推進 **活動（activities）**（步驟）中，其結果只被記錄一次，之後便從日誌中重播。

組成要件：
- **活動（Activities）** 是副作用與非確定性唯一存在的地方；每一個都會被獨立地重試。
- **剛好一次的活動（Exactly-once activities）** 使用冪等鍵（idempotency keys），通常衍生自工作流與步驟的 ID，這樣一次被重試的工具呼叫就不會重複執行。
- **持久化計時器（Durable timers）** 會被持久化，並能熬過 worker 重啟與部署，因此一個工作流可以等待數天，而不必讓一個行程持續開著。
- **訊號（Signals）** 把外部事件（一項核准、一次取消）推入一個運行中的工作流；與之搭配的 **查詢（queries）** 則在不修改其狀態的情況下讀取它當前的狀態（狀態、監控）。當一個工作流在等待訊號或計時器時，worker 會閒置，且在事件抵達之前不消耗任何運算資源，事件抵達時它便重播歷史並恢復。

基於重播的確定性所付出的代價是 **版本控制（versioning）**：因為長時間運行的工作流會重播舊的歷史，所以變更工作流程式碼可能破壞重播並造成事故，除非非常謹慎地做好版本控制。這是最常被提及的單一運維風險。

## 工具

| 工具 | 狀態存放在哪裡 | 佔用規模 | 備註 |
|------|-------------------|-----------|-------|
| **Temporal**（參考標準） | 一個獨立的叢集（或 Temporal Cloud） | 高 | 事件歷史加上確定性重播；支援多種語言；在大規模下已獲驗證；與代理框架的整合最為深入。 |
| **Restate** | 輕量引擎，sidecar 或內嵌 | 低 | 為每個步驟記錄日誌；新增了 Virtual Objects（以使用者／工作階段為鍵的有狀態工作階段，並具備自動並行控制）。 |
| **DBOS** | Postgres 資料列 | 最低 | 一個你匯入的函式庫；工作流狀態與交易型副作用可以共用同一個 Postgres 交易，以在資料庫步驟上達成剛好一次；不需要獨立的叢集。 |
| **Inngest** | 託管、事件驅動 | 低 | 可獨立重試的步驟，具備 AI 專用的基本元件，以及針對 LLM 速率限制內建的並行與節流控制。 |
| **AWS Step Functions** | AWS 託管 | 託管 | 以宣告式狀態機（而非一般程式碼）形式表達的工作流；近期新增了代理運行環境的整合。 |

這些做法的差異在於它們把持久化的邊界放在哪裡。Temporal 用運維開銷換取規模；DBOS 把持久化收進你既有的資料庫；Restate 以 HTTP/gRPC 為原生並具備持久化工作階段；Inngest 是事件驅動且以 TS 為先；Step Functions 以 AWS 為原生，但採宣告式而非通用的程式碼。

## 把持久化執行對應到代理迴圈

核心對應關係是：**代理迴圈變成一個工作流，而每一次模型呼叫與工具呼叫都變成一個持久化活動。** 在崩潰時，已完成的模型呼叫與工具呼叫會從日誌重播，而不是重新執行，因此你不會重新支付 token，也不會重新觸發副作用，你甚至可以修掉一個 bug 並恢復一個運行中的應用。

2026 年的整合版圖：
- **Temporal + OpenAI Agents SDK** 在 2026 年初達到正式可用（general availability），把每一次代理調用與工具呼叫都包裝成一個持久化活動。
- **Temporal + Google ADK** 屬於實驗性質，將 LLM 呼叫重新導向經過活動，並且因為只需要極少的程式碼變更而值得一提（這些包裝器會偵測自己是否運行在工作流內部，否則便退回直接執行）。
- **LangGraph** 透過 **檢查點器（checkpointers）** 提供更輕量、框架原生的持久化，將圖（graph）的狀態在每個 super-step 保存到持久化儲存，並具備可選的持久化模式（在退出時建立檢查點、非同步地，或在每個步驟之前同步地）。它的指引呼應了那條確定性規則：保持工作流的確定性與冪等性，並把副作用包裝在任務（tasks）中。
- **DBOS 與 Restate** 在函式庫層級與代理框架整合，把代理運行與子代理呼叫包裝成持久化工作流與子工作流。

要誠實傳達的張力是：**框架原生的檢查點機制恢復的是 *狀態*；而一個完整的持久化執行引擎額外提供剛好一次的副作用、持久化計時器、訊號，以及跨部署的重播語義。** 當工具呼叫帶有不可逆的外部效應時，這道落差最為要緊。對於那些大多是 LLM 推理、且工具可恢復又冪等的代理而言，框架檢查點加上在少數幾個非冪等工具上的冪等鍵，往往就已經足夠。

把這一切串起來的典型模式是：當一個被提議的動作具有風險時，工作流會暫停並透過訊號等待人類核准，在等待期間不消耗任何運算資源，然後持久化地恢復，這正是 [人類介入（human-in-the-loop）](08-human-in-the-loop-patterns.md) 的核准關卡，只是被打造成防崩潰的版本。

## 何時你會需要它

持久化執行正是 *生產環境* 代理可靠性逐漸浮現的答案，而 2026 年的聲勢是實實在在的：Temporal 以據報達數十億美元的估值募得了一輪龐大的 Series D，且多項重要的 AI 產品都建構於其上。一份被廣泛引用的廠商案例研究描述了一個深度研究代理在遭遇競態條件、脆弱的自製重試邏輯，以及變得難以支援的陳舊狀態 bug 之後，**從一個框架原型遷移到一個持久化執行引擎**（這是一份廠商發布的說法，因此請把方向視為真實，把其框定的角度視為他們自己的）。

但這是一項刻意為之的複雜度取捨。那些限制（確定性、版本控制風險、一套全新的測試與監控模型）都是真實存在的，而對於那些大多唯讀、短命，或單發（single-shot）的代理而言，**框架原生的檢查點機制，或一個佇列加上冪等鍵，往往就已足夠**，而且運維成本低廉得多。相較於一整個叢集，DBOS 與 Restate 大幅降低了進入門檻，所以如果你的顧慮是運維開銷，那麼「函式庫加上 Postgres」的做法或許能取得其中大部分的價值。

## 你需要持久化執行嗎？

依序走過這些問題：

1. **是否有任何工具呼叫帶有不可逆的外部副作用**（付款、寄信、部署、開工單、跨系統寫入）？否：框架檢查點或一個重試／佇列很可能就足夠了。是：繼續。
2. **單次運行是否可能比你的行程或部署週期更長命，或是否必須跨重啟暫停以等待人類核准？** 否：在記憶體中運行，並在完成時建立一個檢查點，大概就沒問題。是：你需要持久化計時器與持久化暫停。
3. **在崩潰時重新運行整個代理是否會無法接受**，無論是在成本、重複效應，或失去長達數小時的進度上？是：你需要重播與剛好一次，因此持久化執行是合理的。
4. **挑選重量級別：** 如果副作用大多是寫入你自己的 Postgres，且你想要一次部署，選 DBOS；若要低運維、HTTP 原生、有狀態工作階段，選 Restate；若要事件驅動、TS 為先、AI 原生的速率限制控制，選 Inngest；如果你全力押注 AWS 且能接受一個宣告式狀態機，選 Step Functions；若要大規模、複雜的長時間運行流程，以及與代理框架最深入的整合，選 Temporal；或者，當代理大多是推理且工具可恢復時，就維持框架原生的持久化（LangGraph 檢查點器）加上冪等鍵。

對於簡單的 CRUD、次毫秒的熱路徑、純粹的高吞吐量串流，或一個需求已經被「一個帶有死信（dead-letter）處理器的佇列」涵蓋的小團隊而言，它都是殺雞用牛刀。

## 面試問題

### Q：為什麼對於一個帶有副作用的生產環境代理來說，天真的重試與檢查點是不夠的？

**有力的回答：**
因為一次代理崩潰製造了一種重試無法安全化解的模糊性。如果代理呼叫了一個會刷卡的工具，然後崩潰，你無法從一份狀態快照判斷這筆扣款是在崩潰前就提交了，還是只有確認訊息遺失了，因此天真的重試有刷兩次卡的風險。一個在步驟之間保存狀態的單純檢查點，仍然在「執行副作用」與「記錄它已發生」之間留下一個不安全的時間窗。持久化執行透過捕捉步驟的日誌（journal）而非只捕捉最新狀態來封閉這道缺口：每一個帶副作用的步驟都是一個帶有冪等鍵的、被記錄下來的活動，所以在重播時，一個已完成的活動會回傳它被記錄的結果，而非重新執行。這為副作用提供了剛好一次的語義，並讓工作流能從它失敗的確切位置恢復，而不是從頭開始。

### Q：何時持久化執行是殺雞用牛刀，而你會改用什麼？

**有力的回答：**
當代理沒有不可逆的副作用、運行得夠短足以塞進一個行程與部署週期之內、且在失敗時直接重新運行也沒問題時，它就是殺雞用牛刀，例如一個帶有冪等工具的唯讀研究或摘要代理。在那種情況下，我會使用框架原生的持久化（例如一個 LangGraph 檢查點器）來恢復狀態，加上在少數幾個非冪等呼叫上的冪等鍵，以及一個帶有死信處理器的重試佇列。像 Temporal 這樣一個完整引擎的確定性限制與版本控制風險是一項真實的成本，所以我只會在代理具有不可逆效應、必須跨重啟暫停以等待人類核准，或長到「在崩潰時重新運行」變得無法接受之後，才承擔它們。如果運維開銷是阻礙，但我仍然需要持久化，那麼像 DBOS 這種使用我既有 Postgres 的函式庫做法，可以在不運行一整個叢集的情況下取得大部分的價值。
